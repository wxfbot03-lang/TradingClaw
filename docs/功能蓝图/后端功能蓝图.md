# TradingClaw 后端服务功能需求文档

## 1. 文档目的
1. 定义系统必须具备的完整业务功能，供团队据此独立开发实现。
2. 明确业务行为、处理规则与验收标准，避免理解偏差。
3. 作为产品、开发、测试三方统一的需求基线。

## 2. 系统定位
### 2.1 系统定位
1. 面向个人投资与策略执行场景，提供行情、交易、策略、资讯与智能分析一体化能力。
2. 同时服务前端应用与自动化任务流程，支持统一会话与业务状态管理。


## 3. 总体业务要求
### 3.1 核心能力要求
1. 必须具备统一应用入口与统一服务编排能力。
2. 必须具备用户体系、资产行情体系、交易体系、策略体系、资讯体系、智能问答体系。
3. 必须具备配置驱动能力，支持模型配置、交易配置、运行参数的动态读取。

### 3.2 安全与合规要求
1. 必须具备请求风险拦截能力，识别并阻断恶意探测与注入型访问。
2. 必须具备认证校验能力，保证用户身份、会话与操作主体一致。
3. 必须具备敏感信息保护能力，日志与响应中不得暴露密钥、令牌、密码。
4. 必须具备基础防护能力，防止参数注入、越权访问等非法请求行为。

### 3.3 稳定性要求
1. 必须具备缓存能力，减少行情和资讯重复拉取。
2. 必须具备失败可感知能力，任何失败都要返回明确的状态与原因。
3. 必须具备服务退化可控能力，在部分外部依赖异常时保持核心能力可用。

## 4. 功能需求明细

### 4.1 用户与会话管理
#### 4.1.1 用户注册登录
1. 提供用户注册，支持基础信息校验与重复用户校验。
2. 提供用户登录，登录成功后生成会话凭证并返回用户基础信息。
3. 提供登录态校验，支持后续业务请求识别用户身份。
4. 提供退出登录，退出后必须使当前会话失效。

#### 4.1.2 用户资料与账户状态
1. 提供用户资料查询，返回等级、额度、积分、已绑定交易账户等信息。
2. 提供基于用户等级的策略可用额度计算。
3. 提供基于策略盈亏的积分占用与可用积分动态计算。
4. 提供用户会话维度的交易账户集合查询。

#### 4.1.3 交易账户管理
1. 支持交易账户绑定关系识别。
2. 支持交易账户解绑。
3. 任何交易操作前必须校验交易账户归属关系。

### 4.2 资产行情与指标服务
#### 4.2.1 实时行情能力
1. 提供股票实时价格查询。
2. 提供基金实时净值查询。
3. 提供股票五档盘口查询。
4. 提供资产关键字搜索，支持证券代码、名称等多条件匹配。

#### 4.2.2 历史行情能力
1. 提供股票历史行情查询，支持按周期与时间范围读取。
2. 提供基金历史净值查询，支持按时间范围读取。
3. 提供历史数据缓存命中、缺口识别、增量补齐能力。
4. 提供历史数据合并、去重、排序与范围截取能力。

#### 4.2.3 技术指标能力
1. 提供股票均线计算，支持多均线类型和多数据点输出。
2. 提供基金均线计算，支持历史窗口与多点计算。
3. 在历史样本不足时必须明确返回不可计算状态。

#### 4.2.4 数据处理规则
1. 必须支持日期格式归一化与交易周期转换。
2. 必须支持默认查询窗口自动推导。
3. 必须支持网络代理策略配置，满足不同部署环境访问外部数据源需求。

### 4.3 新闻资讯与影响力分析
1. 提供多数据源新闻快讯聚合能力。
2. 提供新闻条目统计能力，支持按条件统计数量。
3. 提供新闻影响力打分提交能力。
4. 提供影响力排序起点计算能力。
5. 提供影响力列表查询能力，用于前端展示与策略参考。
6. 新闻能力需支持缓存与分页读取，避免重复抓取。

### 4.4 证券交易能力
#### 4.4.1 交易登录与状态
1. 提供验证码获取能力。
2. 提供手动登录与自动登录能力。
3. 提供交易所会话状态查询能力。
4. 登录失败时需返回明确失败原因，便于用户处理。

#### 4.4.2 账户资产能力
1. 提供综合资产查询。
2. 提供股票持仓查询。
3. 提供基金持仓查询。

#### 4.4.3 下单与撤单能力
1. 提供股票下单。
2. 提供基金下单。
3. 提供股票撤单。
4. 下单与撤单需返回业务执行结果与失败原因。

#### 4.4.4 交易流水能力
1. 提供股票委托明细查询。
2. 提供股票当日委托查询。
3. 提供股票当日成交查询。
4. 提供股票未完成委托查询。
5. 提供基金历史成交查询。

#### 4.4.5 申购能力
1. 提供新股申购能力。
2. 提供新债申购能力。

### 4.5 数字资产交易能力
#### 4.5.1 连接与账户
1. 提供密钥有效性校验。
2. 提供币种持仓查询。
3. 提供币种余额查询。

#### 4.5.2 行情与分析
1. 提供币价查询。
2. 提供历史行情查询。
3. 提供均线指标计算。
4. 提供资金费率查询。
5. 提供成交流水查询。

#### 4.5.3 普通订单能力
1. 提供订单创建。
2. 提供订单修改。
3. 提供订单撤销。
4. 提供订单状态查询。
5. 提供待成交订单查询。

#### 4.5.4 网格交易能力
1. 提供网格策略创建。
2. 提供网格策略停止。
3. 提供网格挂单查询。
4. 提供网格详情查询。
5. 提供网格子单查询。
6. 提供网格改单能力。
7. 提供网格最小投入测算。
8. 提供网格最大下单量测算。
9. 提供网格调仓能力。
10. 提供网格收益提取能力。

### 4.6 策略管理与执行引擎
#### 4.6.1 策略生命周期管理
1. 提供策略创建。
2. 提供策略列表查询。
3. 提供策略详情查询。
4. 提供策略配置修改。
5. 提供策略运行更新。
6. 提供策略停止。
7. 提供策略重启。

#### 4.6.2 策略类型支持
1. 支持股票网格策略。
2. 支持股票马丁策略。
3. 支持股票动态平衡策略。
4. 支持股票基础策略。
5. 支持基金动态平衡策略。
6. 支持基金定投策略。
7. 支持策略完结归档状态管理。

#### 4.6.3 策略计算能力
1. 提供网格价格带生成能力。
2. 提供网格订单生成能力。
3. 提供网格资产占用测算能力。
4. 提供成交后网格重建能力。
5. 提供网格投资需求测算能力。
6. 提供网格限价能力测算。
7. 提供平衡策略可用现金测算。
8. 提供平衡策略调仓单生成。
9. 提供年化收益计算。
10. 提供定投触发条件校验。
11. 提供定投仓位管理计算。

#### 4.6.4 策略公共校验
1. 策略操作必须校验用户登录状态。
2. 策略操作必须校验用户令牌合法性。
3. 策略操作必须校验用户与请求主体一致性。
4. 策略操作必须校验可用策略额度。
5. 策略操作必须校验交易账户归属关系。

### 4.7 统一交易抽象层
1. 提供统一交易能力模型，屏蔽不同交易通道差异。
2. 提供交易通道适配器注册机制，支持扩展新通道。
3. 策略层必须通过统一抽象访问交易行为，不直接依赖具体通道实现。
4. 统一抽象层需覆盖下单、撤单、查单、持仓、成交等核心动作。

### 4.8 智能问答与工具调用
#### 4.8.1 智能能力
1. 提供智能问答能力，支持普通问答与投资场景问答。
2. 支持结合策略、股票、基金上下文进行增强回答。
3. 提供可配置模型设置，支持模型参数持久化与读取。
4. 提供支持模型列表查询，供前端动态展示。

#### 4.8.2 OCR 与辅助能力
1. 提供验证码图片识别能力。
2. 识别结果需返回标准状态与可读错误信息。

#### 4.8.3 工具扩展能力
1. 支持多工具协同调用，用于复杂问题分步推理。
2. 支持后续新增工具接入，保持主流程稳定。

### 4.9 应用入口与基础能力
1. 提供统一入口，集中注册业务能力。
2. 提供统一日志初始化能力。
3. 提供统一响应清洗能力，减少无效空字段输出。

### 4.10 交易计划能力
#### 4.10.1 计划主流程能力
1. 提供交易日维度计划列表查询能力，支持按交易日、状态、市场、标的关键字筛选。
2. 提供计划详情查询能力，返回计划身份(并非登录用户的身份角色)、当前版本、关键位、状态、最新风控结论与校验状态。
3. 提供计划创建能力，支持同日同标的唯一计划身份建立与初始版本初始化。
4. 提供计划版本更新能力，仅允许在草稿或评审中状态修改业务字段。
5. 提供计划情景批量替换能力，支持版本下 1..N 情景重建与排序。
6. 提供基于当前版本创建下一版本能力，用于已批准或已发布计划的修订。

#### 4.10.2 状态机与评审发布能力
1. 提供计划状态迁移能力，支持草稿、评审中、已批准、已发布、已归档状态流转。
2. 状态迁移必须先写状态日志，再变更计划状态，保证审计链路完整。
3. 提供风险评审与复核记录能力，支持结论、风险等级、仓位决策、执行要点等结构化沉淀。
4. 提供发布可用性检查能力，给出评审是否就绪、校验是否就绪、阻塞原因列表。
5. 提供发布提交能力，发布时必须在同一事务内写发布记录、状态日志、状态更新。
6. 发布前必须满足“最新风险评审有效 + 最新校验通过且命中当前内容版本”。

#### 4.10.3 校验与质量能力
1. 提供手工触发校验能力，按目标版本执行规则引擎并记录运行结果。
2. 提供校验问题明细能力，支持错误、警告、提示分级输出。
3. 提供最新校验结果与校验历史查询能力，支持发布页和质量看板使用。
4. 提供质量概览能力，支持按时间范围统计通过率、问题分布、风险规则命中情况。
5. 校验结果必须与版本内容更新时间绑定，避免“旧校验结果误用于新内容发布”。

#### 4.10.4 执行记录与复盘能力
1. 提供信号触发记录能力，用于记录情景触发但未成交场景。
2. 提供执行订单记录能力，支持计划、版本、情景、交易上下文的一致性校验。
3. 执行订单需支持幂等保护，同一交易上下文下重复订单引用不得重复写入。
4. 提供持仓快照与权益快照写入能力，支撑日终复盘与仓位核验。
5. 提供复盘聚合查询能力，汇总版本演进、信号、订单、持仓、权益与执行偏差摘要。

#### 4.10.5 数据模型与约束能力
1. 采用数据库作为交易计划唯一写入事实源，读写围绕计划主表、版本表、情景表、评审表、校验表、执行表、发布表组织。
2. 统一使用整数口径存储百分比、金额、价格，避免浮点误差与跨端口径不一致。
3. 计划版本遵循“业务内容不可变语义”，已批准、已发布、已归档版本禁止原地改写业务内容。
4. 数据库需内建关键约束与触发器，覆盖单计划单 current 版本、合法状态迁移、发布前置条件、金额一致性等规则。
5. 删除策略以逻辑归档为主，计划、评审、执行、发布链路默认禁止物理级联删除。

## 5. 非功能需求
### 5.1 性能要求
1. 高频查询场景必须使用缓存降低外部依赖压力。
2. 数据查询响应时间需满足前端交互级体验要求。
3. 外部源失败时应优先返回缓存可用数据并标注状态。

### 5.2 安全要求
1. 必须具备恶意请求拦截策略。
2. 必须具备鉴权失败快速返回能力。
3. 必须具备敏感信息脱敏与最小暴露原则。

### 5.3 可维护性要求
1. 业务模块边界清晰，职责单一。
2. 公共能力复用统一组件，避免重复实现。
3. 新增交易通道与策略类型时不影响既有业务能力。

### 5.4 可观测性要求
1. 关键业务动作必须有日志埋点。
2. 异常必须记录上下文，支持问题追踪。
3. 需支持按模块定位故障来源。

## 6. 验收标准
1. 本文档列出的全部功能点均可实现并可被测试验证。
2. 用户、行情、交易、策略、资讯、智能问答能力全部可用。
3. 交易计划能力完成第一阶段最小可用闭环：创建、编辑、状态迁移、评审、校验、发布。
4. 安全校验与权限校验在所有关键入口生效。
5. 交易计划执行与复盘能力可完成信号记录、订单记录、快照写入与复盘查询。
6. 缓存、日志、异常处理等通用能力在全链路可观测。
