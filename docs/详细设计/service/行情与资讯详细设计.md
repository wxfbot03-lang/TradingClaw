## 2. 模块职责

### 2.1 `market-service`

- 股票、基金、数字资产实时行情查询。
- 历史行情、K 线、均线等指标供给。
- 行情标准化、缓存回退、缺口识别与增量补齐。
- 持有 `Instrument` 主数据与 `instrument_id` 唯一标识口径，供交易、策略、事件链路统一引用。

### 2.2 `news-service`

- 多源资讯抓取、聚合、去重、分页查询。
- 资讯影响力评分与排序。

## 5. 核心业务流程

标的主权说明：

- `market-service` 是 `Instrument` 与 `instrument_id` 的唯一 owner。
- 交易、策略、AI、事件消费者只能引用 `instrument_id` 或 `symbol + market`，不得在本域重建独立标的主数据字典。
- 标准接口、事件分区键、缓存主键应优先使用 `instrument_id`；`symbol + market` 仅作为兼容定位键。

### 5.1 实时行情查询

1. 接收行情查询请求。
2. 优先从缓存读取热点数据。
3. 未命中时访问外部行情源。
4. 标准化后写回缓存并返回。
5. 外部依赖失败时回退可用缓存并标记数据状态。

数据源治理要求：

- 必须明确首批供应商、主备优先级和故障切换策略。
- 多源返回冲突时，按来源优先级、时间戳新鲜度和可信度评分合并。
- 实时行情默认新鲜度 SLA 建议不超过 2 秒；超出阈值必须返回降级标识。

### 5.2 历史行情补齐

1. 接收区间查询或离线回填任务。
2. 检查已有区间和缺口。
3. 通过 `HistoricalBackfillWorkflow` 补齐缺口。
4. 完成去重、排序、标准化后写入时序库。
5. 发布 `historical.backfilled` 事件。

### 5.3 指标计算

1. 读取标准化 K 线序列。
2. 根据指标类型和窗口进行计算。
3. 样本不足时返回不可计算状态。
4. 计算结果进入分析库并可发布 `indicator.calculated` 事件。

### 5.4 资讯聚合与评分

1. 从多个资讯源抓取原始新闻。
2. 标准化、清洗并建立去重指纹。
3. 入库标准资讯实体。
4. 执行影响力评分。
5. 写入检索索引并发布 `news.ingested`、`news.scored` 事件。

资讯治理要求：

- 首批资讯供应商必须列清授权范围、抓取频率和优先级。
- 同题多源资讯需按标题指纹、发布时间窗口和来源等级合并，避免重复刷屏。
- 评分模型版本、数据源冲突处理规则和延迟目标必须可审计。

## 6. 数据设计

事务型元数据表：

- `instruments`
- `market_data_sources`
- `news_items`
- `news_dedup_keys`
- `news_scores`

分析型逻辑表：

- `ticks`
- `klines`
- `indicator_series`

设计要点：

- 行情明细和 K 线不堆在事务库。
- 资讯正文可在事务库保留结构化摘要，在搜索引擎保留检索视图。
- `instruments`、`market_data_sources`、`news_items`、`news_dedup_keys`、`news_scores` 等元数据和资讯主记录可落 MySQL。
- Redis 仅用于热点行情、指标查询结果、资讯列表短缓存和回退缓存；缓存失效后必须能从时序/分析库、MySQL 或外部源回补。
- 默认值建议：`instruments.status = ACTIVE`，`news_scores.score` 无默认值且必须由评分任务写入。
- 空值规则：`summary`、`content_ref`、`reason` 可空；`instrument_id`、`symbol`、`market`、`asset_class`、`news_id`、`published_at` 默认不可空。
- 删除策略：标的主数据与资讯主记录默认不物理删除，通过状态失效、去索引或归档处理；高频时序数据按保留周期滚动清理。
- 审计要求：资讯评分、来源配置变更建议记录 `source_id`、`model_version`、`updated_by`。
- 时序库选型、对象存储和搜索索引方案必须在实现前冻结；MVP 不建议边开发边替换底层存储。

### 6.1 `instruments`

| --- | --- | --- | --- |

| `symbol` | varchar(64) | UK(symbol, market) | 标的代码 |
| `market` | varchar(32) | UK(symbol, market) | 市场代码 |
| `asset_class` | varchar(32) | idx | 资产类别 |
| `name` | varchar(128) |  | 标的名称 |
| `status` | varchar(32) | idx | 标的状态 |
| `price_precision` | integer |  | 价格精度 |
| `quantity_precision` | integer |  | 数量精度 |
| `updated_at` | datetime | idx | 更新时间 |

### 6.2 `news_items`

| 字段 | 类型建议 | 约束/索引 | 说明 |
| --- | --- | --- | --- |
| `id` | bigint / uuid | PK | 主键 |
| `news_id` | varchar(64) | UK | 资讯 ID |
| `source_id` | varchar(64) | idx | 来源 ID |
| `title` | varchar(255) |  | 标题 |
| `summary` | text |  | 摘要 |
| `published_at` | datetime | idx | 发布时间 |
| `dedup_key` | varchar(128) | idx | 去重键 |
| `content_ref` | varchar(255) |  | 正文引用 |

### 6.3 `news_scores`

| 字段 | 类型建议 | 约束/索引 | 说明 |
| --- | --- | --- | --- |
| `id` | bigint / uuid | PK | 主键 |
| `news_id` | varchar(64) | UK(news_id, model_version) | 资讯 ID |
| `model_version` | varchar(64) | UK(news_id, model_version) | 评分模型版本 |
| `score` | decimal(10,4) | idx | 影响力评分 |
| `reason` | text |  | 评分说明 |
| `scored_at` | datetime | idx | 评分时间 |

### 6.4 `ticks` / `klines` / `indicator_series`

- 三类高频行情表属于时序/分析存储，不建议直接落 MySQL 明细表。
- 主键建议由 `instrument_id + period + timestamp/bar_time` 组合确定，`symbol + market` 仅保留查询兼容能力。
- Redis 仅缓存热点窗口结果，不能替代时序主存。

#### 8.1.1 `GET /api/v1/market/quotes`

查询参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `instrument_ids` | string | 否 | 推荐使用，多个标的 ID 以逗号分隔 |
| `symbols` | string | 否 | 兼容模式下多个标的代码以逗号分隔 |
| `market` | string | 否 | 市场代码 |
| `with_depth` | boolean | 否 | 是否返回五档盘口 |

语义约束：

- 标准模式优先使用 `instrument_ids`。
- 仅当存量调用方尚未完成改造时，允许使用 `symbols + market`。

返回体 `data`：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `quotes` | array | 行情列表 |
| `quotes[].instrument_id` | string | 标的统一唯一标识 |
| `quotes[].symbol` | string | 标的代码 |
| `quotes[].price` | number | 最新价 |
| `quotes[].timestamp` | string | 数据时间 |
| `quotes[].source` | string | 数据源 |
| `quotes[].depth` | array | 五档盘口，可选 |
| `data_status` | string | `fresh`、`cache_fallback`、`partial` 或 `unavailable` |

## 9. 依赖与实施顺序

- 本模块建议第二批实现。
- 是策略系统、AI、交易查询的重要基础数据依赖。
- 行情标准化口径必须在本模块统一，不允许策略和交易模块各自做二次定义。
