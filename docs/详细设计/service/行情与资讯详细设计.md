# TradingClaw 行情与资讯详细设计

## 1. 文档定位

- 本文档覆盖 `market-service` 与 `news-service`，定义标的主数据、实时行情、历史行情、指标计算和资讯聚合的统一设计。
- 本模块为交易、策略、AI 和运营治理提供标准化市场数据与资讯供给能力。
- `market-service` 独占 `instrument_id` 主权，其他领域只能引用，不得重建独立标的主数据口径。

## 2. 模块定位

### 2.1 `market-service`

- 维护统一标的主数据和 `instrument_id`。
- 提供实时行情、历史行情、K 线和指标供给。
- 负责行情标准化、缓存回退、缺口识别和增量回补。

### 2.2 `news-service`

- 提供多源资讯抓取、标准化、去重、评分和分页查询。
- 为 AI、策略和运营提供资讯检索与影响力排序能力。

## 3. 领域边界

- `market-service` 是 `Instrument` 与 `instrument_id` 的唯一 owner。
- 交易、策略、AI 和事件消费者应优先使用 `instrument_id` 作为查询键、缓存键和分区键，`symbol + market` 仅保留兼容定位能力。
- 行情与资讯服务输出标准化事实，不负责交易执行、策略状态推进或 AI 编排结果。
- 行情高频明细进入时序或分析存储，不以事务库承担主吞吐。

## 5. 市场数据与资讯主线

### 5.1 标的主数据主线

标的主数据是市场域对外输出的基础主锚点。

约束如下：

- `instrument_id` 是跨域稳定主键，由市场域独占生成和维护。
- 交易、策略、AI、事件链路统一引用 `instrument_id`，不得在其他领域重新生成主标识。
- `symbol + market` 只作为兼容定位键，不得替代 `instrument_id` 成为标准主键。

### 5.2 实时行情主线

流程如下：

1. 接收实时行情查询请求。
2. 优先从热点缓存读取标准化行情结果。
3. 缓存未命中或过期时访问外部行情源。
4. 对多源返回结果执行标准化、冲突合并和时间新鲜度判断。
5. 写回缓存并返回查询结果；若外部源不可用，则回退最近可用缓存并标记 `data_status`。

治理要求：

- 必须明确首批行情供应商、主备优先级和故障切换策略。
- 多源冲突时，按来源优先级、时间戳新鲜度和可信度评分合并。
- 实时行情默认新鲜度 SLA 建议不超过 2 秒；超出阈值必须返回降级标识。

### 5.3 历史行情与回补主线

流程如下：

1. 接收区间查询请求或离线回填任务。
2. 检查目标区间已有数据和缺口。
3. 通过 `HistoricalBackfillWorkflow` 补齐缺口。
4. 完成去重、排序和标准化后写入时序库。
5. 发布 `historical.backfilled` 事件供下游感知数据补齐结果。

### 5.4 指标计算主线

流程如下：

1. 读取标准化 K 线序列。
2. 根据指标类型、窗口和周期执行计算。
3. 样本不足时返回 `calculable = false` 与结构化原因。
4. 计算结果写入分析存储，并按需发布 `indicator.calculated` 事件。

### 5.5 资讯聚合与评分主线

流程如下：

1. 从多个资讯源抓取原始新闻。
2. 标准化、清洗并建立去重指纹。
3. 入库标准资讯实体。
4. 执行影响力评分，记录模型版本与评分说明。
5. 写入检索索引，并发布 `news.ingested`、`news.scored` 事件。

治理要求：

- 首批资讯供应商必须列清授权范围、抓取频率和优先级。
- 同题多源资讯需按标题指纹、发布时间窗口和来源等级合并，避免重复刷屏。
- 评分模型版本、数据源冲突处理规则和延迟目标必须可审计。

## 6. 数据设计

事务型元数据表：

- `instruments`
- `market_data_sources`
- `news_items`
- `news_dedup_keys`
- `news_scores`

分析型逻辑表：

- `ticks`
- `klines`
- `indicator_series`

设计要点：

- `instruments`、`market_data_sources`、`news_items`、`news_dedup_keys`、`news_scores` 等元数据和资讯主记录可落 MySQL。
- 高频行情明细、K 线和指标序列不应堆在事务库，应进入时序或分析存储。
- Redis 仅用于热点行情、指标查询结果、资讯列表短缓存和回退缓存；缓存失效后必须可从时序库、MySQL 或外部源回补。
- 资讯正文可在事务库保留结构化摘要，在搜索引擎保留检索视图，对象存储保留长文本或原文引用。
- 默认值建议：`instruments.status = ACTIVE`；`news_scores.score` 无默认值，必须由评分流程写入。
- 标的主数据与资讯主记录默认不物理删除，通过状态失效、去索引或归档处理；高频时序数据按保留周期滚动清理。
- 时序库、搜索引擎和对象存储方案必须在实现前冻结。

| 字段 | 类型建议 | 约束/索引 | 说明 |

| `news_id` | varchar(64) | UK(news_id, model_version) | 资讯 ID |

| `scored_at` | datetime | idx | 评分时间 |

### 6.4 `ticks` / `klines` / `indicator_series`

- 三类高频行情表属于时序或分析存储，不建议直接落 MySQL 明细表。
- 主键建议由 `instrument_id + period + timestamp/bar_time` 组合确定，`symbol + market` 仅保留查询兼容能力。
- Redis 只缓存热点窗口结果，不能替代时序主存。

## 7. 事件设计

核心事件：

- `quote.updated`
- `kline.closed`
- `historical.backfilled`
- `indicator.calculated`
- `news.ingested`
- `news.scored`

事件约束：

- 行情事件和补齐事件优先以 `instrument_id` 作为分区键。
- 资讯事件优先以 `news_id` 作为聚合锚点。

## 9. 实施要点

- 本模块建议第二批实现，是交易、策略、AI 和运营分析的重要基础依赖。
- 行情标准化、标的主键和资讯评分口径必须在本模块统一，不允许交易和策略模块各自做二次定义。
- 实现上建议使用 FastAPI 承载查询接口，时序计算和历史回补通过工作流与后台任务编排完成。

| 资讯抓取 | 更新检测能力 | 感知资讯修订、撤稿或删除 | 用于纠错和索引维护 |
| 订阅能力 | 新资讯推送/回调 | 实时接收新资讯 | 可降低轮询抓取成本 |

#### 10.2.3 可选补充能力

补充说明：

- 行情接入最低可用闭环通常需要：标的主数据、最新价、历史 K 线、区间查询、时间戳和限流说明。
