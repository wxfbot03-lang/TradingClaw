# TradingClaw 错误码字典

## 1. 文档定位

- 本文档统一 `docs/详细设计/service/` 下各模块详细设计中的错误码口径。
- 目标是保证网关、领域服务、事件消费、工作流活动和审计回放在错误语义上可映射、可追踪、可审计。
- 本文档定义错误码分层、命名规则、公共错误码和各领域错误码建议，是 `reason_code` 字段的统一来源之一。

## 2. 设计原则

- 错误码必须稳定，不因内部实现变化而频繁变更。
- 错误码与可读提示分离；错误码用于机器处理，`message` 用于人类阅读。
- 网关对外返回的错误码必须可由内部错误语义稳定映射。
- 事件与工作流中的失败也必须区分业务拒绝和系统故障。
- 同一错误语义禁止在多个模块使用不同错误码。

## 3. 编码规则

统一格式：

```text
<DOMAIN>-<SCENARIO>-<NUMBER>
```

示例：

- `GW-AUTH-001`
- `AUTH-LOGIN-002`
- `TRD-ORDER-004`
- `STR-STATE-003`

规则说明：

- `DOMAIN`：错误所属域，如 `GW`、`AUTH`、`TRD`。
- `SCENARIO`：错误场景，如 `AUTH`、`ORDER`、`STATE`、`RISK`。
- `NUMBER`：三位流水号。

## 4. 错误码分层

| 域前缀 | 模块 | 说明 |
| --- | --- | --- |
| `GW` | 网关与平台基础 | 参数校验、鉴权入口、限流、能力未开放 |
| `AUTH` | 用户与会话 | 登录、会话、权限、身份断言 |
| `ACC` | 账户与归属 | 账户绑定、归属、默认会话、能力矩阵 |
| `MKT` | 行情与指标 | 行情查询、历史补齐、指标计算 |
| `NEWS` | 资讯与评分 | 资讯抓取、去重、评分、分页 |
| `TRD` | 统一交易 | 下单、撤单、订单状态、持仓、余额 |
| `SEC` | 证券交易 | 券商登录、验证码、券商回报 |
| `CRY` | 数字资产交易 | 密钥校验、交易所适配、资金费率 |
| `STR` | 策略系统 | 策略配置、状态、快照、运行时 |
| `AI` | 智能能力 | 模型配置、问答、工具调用 |
| `OCR` | OCR | 图片识别、任务失败 |
| `CAP` | 能力治理 | 版本矩阵、能力未开放、能力冲突 |
| `RSK` | 风控 | 风险命中、复核要求、裁决拒绝 |
| `AUD` | 审计 | 审计写入、审计查询 |
| `NTF` | 通知 | 通知创建、投递失败 |
| `SYS` | 平台系统 | 超时、依赖失败、配置错误、内部故障 |

## 11. 错误响应结构建议

```json
{
  "request_id": "req_001",
  "code": "TRD-ORDER-004",
  "message": "channel rejected order",
  "data": {},
  "meta": {
    "trace_id": "trc_001",
    "reason_code": "TRD-ORDER-004"
  }
}
```

建议附加字段：

- `meta.field_errors`：字段级校验错误
- `meta.capability_status`：能力未开放时的版本说明
- `meta.retryable`：是否建议调用方重试

## 12. 映射规则

- 内部模块错误码可以更细，但必须稳定映射到统一对外错误码。
- 事件消费失败不直接返回 HTTP 错误码，但必须保留统一错误码语义。
- 工作流活动失败需要区分业务失败和系统失败，并映射到统一错误码域。
- 通道透传错误信息只用于排障，不直接充当统一业务错误码。

## 13. 使用规则

- 新增模块错误码时，优先在所属域前缀下扩展，不新增随意前缀。
- 模块文档中的 `reason_code` 字段取值必须引用本字典。
- 若错误语义已存在，应复用现有错误码，而不是重新编号。
