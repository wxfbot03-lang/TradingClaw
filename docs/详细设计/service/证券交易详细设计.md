# TradingClaw 证券交易详细设计

## 1. 文档定位

- 本文档覆盖 `securities-trading-service`，定义券商适配域的认证、会话、原始查询、原始下单和回报同步设计。
- 本模块负责券商协议适配，不直接对外暴露统一交易语义。
- 统一订单状态、统一错误码、统一交易会话和统一订单事实均由交易网关维护，本模块只输出适配侧事实。

## 2. 模块定位

- 负责券商认证、挑战材料获取、会话保持与刷新。
- 负责证券账户资产、持仓、委托、成交等原始查询。
- 负责股票、基金等证券品类的券商协议下单与撤单。
- 将券商原始状态和原始回报标准化为适配侧输出，再由交易网关归一为统一交易事实。

## 3. 领域边界

- 对上游只暴露通道适配能力，不直接面向前端、CLI 或 AI。
- 原始查询接口仅供 `trade-gateway-service`、工作流活动和运维排障使用，不得被客户端或策略运行时直接依赖。
- 本模块不发布统一语义的 `trading_session.*` 事件，不维护统一订单状态机。
- 本模块不得直接写入账户域能力矩阵或交易网关主表，只能通过 gRPC 返回结果或事件传播适配事实。

## 5. 券商适配主线

### 5.1 证券适配域的接入模型

证券适配域当前优先覆盖需要券商网页或 HTTP 登录态的接入方，但设计上不把“证券”与“交互登录”硬绑定。是否需要多步登录、验证码、短信验证或签名鉴权，应由接入方元数据决定，而不是由账户类别决定。

本模块当前优先支持的典型模式是：

- `adapter_auth_mode = INTERACTIVE_LOGIN`
- `adapter_session_mode = STATEFUL_SESSION`
- `challenge_mode = IMAGE_CAPTCHA`、`SMS_OTP` 或 `NONE`

例如东方财富可以采用：`provider_code=eastmoney`、`adapter_auth_mode=INTERACTIVE_LOGIN`、`adapter_session_mode=STATEFUL_SESSION`。若未来某家券商提供签名型 API，则仍可继续纳入证券适配域，只需切换对应的认证和会话模式。

### 5.2 凭据模型与步骤输入边界

`credential_ref` 在证券适配域中表达的是稳定存在的接入凭据引用，而不是一次登录过程中所有输入的集合。

交互登录型凭据通常至少包含：

| 字段 | 说明 |
| --- | --- |
| `provider_code` | 接入方标识，如 `eastmoney` |
| `login_principal` | 登录账号 |
| `login_password` | 登录密码 |
| `extra_login_context` | 营业部、设备号、登录域等固定附加上下文 |
| `secret_schema` | 凭据结构标识，如 `username_password_captcha_v1` |

边界约束：

- 真实账号密码必须由密钥管理系统保存，业务库只保存 `credential_ref` 和元数据。
- 图片验证码识别结果、短信验证码、OTP、challenge token 等动态材料都属于步骤输入，不应持久化为长期凭据字段。
- 适配域应根据 `auth_flow` 当前步骤向交易网关返回 `required_fields` 与 `artifacts`，而不是要求前端预先知道所有券商差异。

### 5.3 券商认证与会话建立主线

券商认证的目标不是返回某个网页登录结果，而是向交易网关证明“当前账户已经建立可继续使用的券商交易上下文”。

流程如下：

1. 交易网关基于 `provider_code + adapter_auth_mode + adapter_session_mode` 调用证券适配域启动认证流程。
2. 证券适配域读取 `credential_ref` 元数据和真实密钥材料，初始化 `BrokerAuthFlow`。
3. 若流程可一步完成，则直接执行登录、账户探测或有效性校验，建立 `BrokerSession` 并返回 `adapter_session_status = AVAILABLE`。
4. 若流程需要用户补充输入，则返回 `auth_flow_id`、`challenge_required = true` 和 `next_step`。
5. 交易网关提交当前步骤输入后，证券适配域继续推进认证流程，直到认证成功、失败或超时。
6. 认证成功后建立或刷新 `BrokerSession`，并通过 gRPC 返回或适配事件把适配侧会话状态同步给交易网关。
7. 对账户能力、可交易市场和校验结果进行标准化，发布 `broker_account.validated` 供账户域更新能力投影。
8. 认证失败时记录结构化原因码，并支持重试、超时失效或人工介入。

统一约束：

- 适配侧会话状态统一使用 `adapter_session_status`，不得对外冒用 `trading_session_status`。
- 交易网关消费 `broker_session.updated` 或同步结果后，才可推进统一交易会话。
- 券商认证流程必须返回“下一步需要什么”，而不是把所有交互都压缩成单次登录接口。

### 5.4 多步认证示例

典型示例一：先输入账号，再输入密码与图片验证码

1. 交易网关启动交易会话认证流程。
2. 证券适配域返回下一步 `SUBMIT_LOGIN_PRINCIPAL`。
3. 用户提交登录账号。
4. 券商返回验证码图片。
5. 证券适配域返回下一步 `SUBMIT_PASSWORD_AND_CAPTCHA`，并在 `artifacts` 中带回验证码图片引用。
6. 用户提交密码与 `captcha_text`。
7. 登录成功后建立 `BrokerSession`。

典型示例二：先输入账号密码，再输入短信验证码

1. 交易网关启动交易会话认证流程。
2. 证券适配域返回下一步 `SUBMIT_LOGIN_CREDENTIALS`。
3. 用户提交账号与密码。
4. 券商发送短信验证码。
5. 证券适配域返回下一步 `SUBMIT_SMS_CODE`，并在 `artifacts` 中带回短信目标信息或提示文案。
6. 用户提交 `sms_code`。
7. 登录成功后建立 `BrokerSession`。

典型示例三：登录后仍需补充 OTP

1. 交易网关启动交易会话认证流程。
2. 券商完成基础账号密码校验，但要求二次验证。
3. 证券适配域返回下一步 `SUBMIT_OTP_CODE`。
4. 用户提交 `otp_code`。
5. 校验通过后建立 `BrokerSession`。

### 5.5 运行时会话语义

证券适配域通常需要真实的运行时 HTTP 会话来维持券商登录态，但运行时容器不是长期主存事实。

实现要求：

- 登录成功后的运行时上下文可由 `requests.Session` 或等价实现承载。
- `requests.Session` 只作为适配域运行时会话容器，不直接作为长期持久化对象写入 MySQL。
- MySQL 中只保留 `broker_session_id`、状态、过期时间、最近错误码、运行时引用和必要审计字段。
- Redis 或进程内缓存可保存短期运行时上下文映射；多实例部署时，应通过共享缓存、会话映射表或粘性路由保证后续请求能找到正确的运行时会话。
- 运行时上下文失效后，必须能通过刷新认证流程重建，而不是依赖不可恢复的内存对象。

### 5.6 原始查询主线

流程如下：

1. 接收来自交易网关的原始查询请求。
2. 校验对应 `BrokerSession` 是否可用；必要时先执行快速续期或健康检查。
3. 调用券商资产、持仓、委托、成交等原始查询接口。
4. 将结果转换为适配域标准对象，返回给交易网关或写入适配侧快照。
5. 对大报文保留原始归档引用，供审计与排障使用。

约束：

- 原始查询接口只返回券商原始视图或适配层标准化结果，不得被上游直接当作统一交易事实。
- 若券商不支持推送，本模块必须提供稳定的轮询查询与状态回补能力。

### 5.7 证券下单与撤单主线

流程如下：

1. 接收来自交易网关的标准化下单或撤单命令。
2. 依据券商能力矩阵、交易规则和当前市场限制映射为券商协议字段。
3. 调用券商委托接口。
4. 返回首个原始结果，并记录原始请求与原始响应引用。
5. 通过回报同步链路持续补充标准化执行回报。

约束：

- 本模块返回的是适配侧原始结果和适配标准化结果，不是统一订单事实。
- “券商原始状态 -> 统一订单状态 -> 展示状态”的最终映射口径必须由统一映射表管理，不允许在多个位置散落转换逻辑。
- 券商侧若存在最小下单单位、盘前盘后限制、可撤状态差异，必须在能力矩阵和规则映射中显式表达。

### 5.8 券商回报同步主线

流程如下：

1. 接收券商主动回调或轮询结果。
2. 将原始状态转换为适配标准化回报对象。
3. 发布 `execution.report_received` 事件。
4. 归档原始报文到对象存储或归档表。

约束：

- 所有券商报文必须保留原始归档。
- 原始状态保存在 `raw_status`，不得覆盖统一状态字段。

## 6. 数据设计

核心表：

- `broker_auth_flows`
- `broker_sessions`
- `securities_assets_snapshots`
- `securities_positions_snapshots`
- `securities_order_records`
- `securities_trade_records`

设计要点：

- 券商认证流程、券商会话、原始委托与成交回报建议落 MySQL，确保适配链路具备审计和问题回放能力。
- `securities_assets_snapshots` 与 `securities_positions_snapshots` 可按查询周期刷新，Redis 只缓存最新可读视图。
- 原始券商请求报文和响应报文建议存对象存储或归档表，MySQL 中仅保留索引字段与引用。
- 默认值建议：`broker_auth_flows.flow_status = IN_PROGRESS`，`broker_sessions.adapter_session_status = AUTHENTICATING`，`securities_order_records.raw_status` 保持原样记录。
- 券商会话、原始订单记录和成交记录默认不物理删除，异常数据通过补录或状态修订处理。

## 9. 实施要点

- 所有券商报文必须保留原始归档，所有状态映射必须集中维护。
- 券商认证失败原因必须结构化记录，供交易网关、前端和 CLI 返回可读提示。
- 步骤式认证实现建议由显式状态机或工作流统一编排，避免把验证码、短信、OTP 等挑战散落为多个一次性接口。
- 必须提供首批券商能力矩阵样例，至少覆盖支持市场、盘前盘后、最小下单单位、撤单限制和可查询字段。
- 必须提供券商错误码到统一错误码的映射细则，未知错误统一落到保底错误族并保留原始错误引用。

## 10. 证券接入 API 能力清单

本节只整理券商侧需要提供的 API 功能能力，不约束其参数和返回值格式；具体接口定义、字段名、错误码和报文结构由券商提供后再做适配映射。

### 10.1 MVP 必需能力

| 类别 | API 功能 | 用途 | 说明 |
| --- | --- | --- | --- |
| 认证接入 | 认证启动 | 启动券商认证流程 | 支持账号登录、令牌登录或等价认证方式 |
| 认证接入 | 会话刷新/续期 | 延长登录态或恢复可用会话 | 用于保持可交易会话 |
| 认证接入 | 挑战材料获取 | 获取验证码图片、短信下发结果或等价挑战材料 | 仅在券商登录依赖挑战时需要 |
| 认证接入 | 认证步骤推进 | 提交验证码、短信码、OTP 或等价输入 | 若挑战已并入登录接口，可由同一接口承担 |
| 账户查询 | 账户基础信息查询 | 获取账户身份、市场权限、账户类型等 | 用于账户校验和能力投影 |
| 账户查询 | 资金资产查询 | 获取总资产、可用资金、冻结资金等 | 用于交易前校验和统一余额读模型回补 |
| 账户查询 | 持仓查询 | 获取当前持仓和可卖数量 | 用于交易校验和持仓读模型回补 |
| 账户查询 | 当日委托查询 | 获取当日委托记录 | 用于查单和状态回补 |
| 账户查询 | 当日成交查询 | 获取当日成交记录 | 用于成交回补和对账 |
| 交易写入 | 证券买入下单 | 提交买入委托 | 股票、基金等基础买入能力 |
| 交易写入 | 证券卖出下单 | 提交卖出委托 | 股票、基金等基础卖出能力 |
| 交易写入 | 撤单 | 对未终结委托发起撤销 | 需能返回撤单受理结果 |
| 交易写入 | 订单状态查询 | 查询单笔委托当前状态 | 用于同步查单、补偿和排障 |
| 回报同步 | 回报轮询查询 | 定时拉取委托与成交增量变化 | 作为无推送场景主方案，或推送失败时兜底 |
| 交易规则 | 错误码说明 | 提供券商错误码语义 | 用于映射统一错误码和可读提示 |
| 稳定性支持 | 限频/限流规则说明 | 明确接口调用频控限制 | 用于适配层限流和重试策略 |

补充说明：

- 最低可接入闭环通常需要：认证启动、挑战推进、资金查询、持仓查询、下单、撤单、订单状态查询、成交查询或成交回补能力。
- 若券商不支持主动回报推送，则必须提供可靠的委托/成交轮询查询能力。
- 若券商不提供下单规则查询接口，则至少需要以文档或配置方式提供规则清单，便于我方维护映射和校验逻辑。
