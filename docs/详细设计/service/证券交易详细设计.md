# TradingClaw 证券交易详细设计

## 1. 文档定位

- 本文档覆盖 `securities-trading-service`，定义券商适配域的登录、验证码、原始查询、原始下单和回报同步设计。
- 本模块负责券商协议适配，不直接对外暴露统一交易语义。
- 统一订单状态、统一错误码、统一交易会话和统一订单事实均由交易网关维护，本模块只输出适配侧事实。

## 2. 模块定位

- 负责券商登录、验证码获取、会话保持与刷新。
- 负责证券账户资产、持仓、委托、成交等原始查询。
- 负责股票、基金等证券品类的券商协议下单与撤单。
- 将券商原始状态和原始回报标准化为适配侧输出，再由交易网关归一为统一交易事实。

## 3. 领域边界

- 对上游只暴露通道适配能力，不直接面向前端、CLI 或 AI。
- 原始查询接口仅供 `trade-gateway-service`、工作流活动和运维排障使用，不得被客户端或策略运行时直接依赖。
- 本模块不发布统一语义的 `trading_session.*` 事件，不维护统一订单状态机。
- 本模块不得直接写入账户域能力矩阵或交易网关主表，只能通过 gRPC 返回结果或事件传播适配事实。

## 5. 券商适配主线

### 5.1 券商登录与会话主线

流程如下：

1. 触发 `BrokerLoginWorkflow`。
2. 获取验证码或执行免验证码认证。
3. 调用券商登录接口。
4. 建立 `BrokerSession`，并通过 gRPC 返回或适配事件把原始会话状态同步给交易网关。
5. 对账户能力、可交易市场和校验结果进行标准化，发布 `broker_account.validated` 供账户域更新能力投影。
6. 登录失败时记录结构化原因码，并支持重试或人工介入。

约束：

- 适配侧会话状态统一使用 `adapter_session_status`，不得对外冒用 `trading_session_status`。
- 交易网关消费 `broker_session.updated` 或同步结果后，才可推进统一交易会话。

### 5.2 证券下单与撤单主线

流程如下：

1. 接收来自交易网关的标准化下单或撤单命令。
2. 映射为券商协议字段和券商下单参数。
3. 调用券商委托接口。
4. 返回首个原始结果，并记录原始请求与原始响应引用。
5. 通过回报同步链路持续补充标准化执行回报。

约束：

- 本模块返回的是适配侧原始结果和适配标准化结果，不是统一订单事实。
- “券商原始状态 -> 统一订单状态 -> 展示状态”的最终映射口径必须由统一映射表管理，不允许在多个位置散落转换逻辑。

### 5.3 券商回报同步主线

流程如下：

1. 接收券商主动回调或轮询结果。
2. 将原始状态转换为适配标准化回报对象。
3. 发布 `execution.report_received` 事件。
4. 归档原始报文到对象存储或归档表。

约束：

- 所有券商报文必须保留原始归档。
- 原始状态保存在 `raw_status`，不得覆盖统一状态字段。

## 6. 数据设计

核心表：

- `broker_login_tasks`
- `broker_captcha_jobs`
- `broker_sessions`
- `securities_assets_snapshots`
- `securities_positions_snapshots`
- `securities_order_records`
- `securities_trade_records`

设计要点：

- 券商登录、验证码、券商会话、原始委托与成交回报建议落 MySQL，确保适配链路具备审计和问题回放能力。
- `securities_assets_snapshots` 与 `securities_positions_snapshots` 可按查询周期刷新，Redis 只缓存最新可读视图。
- 原始券商请求报文和响应报文建议存对象存储或归档表，MySQL 中仅保留索引字段与引用。
- 默认值建议：`broker_captcha_jobs.status = CREATED`，`broker_sessions.adapter_session_status = AUTHENTICATING`，`securities_order_records.raw_status` 保持原样记录。
- 券商会话、原始订单记录和成交记录默认不物理删除，异常数据通过补录或状态修订处理。

## 7. 事件设计

核心事件：

- `execution.report_received`
- `broker_session.updated`
- `broker_account.validated`
- `notification.send_requested`

事件约束：

- `notification.send_requested` 仅作为发给通知域的集成命令事件，不是通知领域事实事件。
- 本模块不发布统一语义的 `trading_session.*` 事件。

## 8. 接口设计

### 8.1 gRPC 服务

- `BrokerLoginService`
- `SecuritiesOrderAdapterService`
- `SecuritiesAssetAdapterService`
- `SecuritiesQueryService`

#### 8.1.1 `BrokerLoginService.Login`

请求字段：`account_id`、`broker_code`、`credential_ref`、`captcha_text`

响应字段：`broker_session_id`、`adapter_session_status`、`reason_code`

#### 8.1.2 `SecuritiesOrderAdapterService.PlaceOrder`

请求字段：`account_id`、`channel_order_request`

响应字段：`channel_order_id`、`raw_status`、`accepted`

#### 8.1.3 `SecuritiesAssetAdapterService.GetAssetsRaw`

请求字段：`account_id`

响应字段：`asset_summary`、`positions`

语义约束：仅供 `trade-gateway-service`、工作流活动和运维排障使用，不作为统一交易查询接口。

#### 8.1.4 `SecuritiesQueryService.GetOrderStatusRaw`

请求字段：`account_id`、`channel_order_id`

响应字段：`raw_status`、`normalized_report`

语义约束：仅返回券商原始视图或适配归一化结果，不得被上游直接当作统一订单事实。

### 8.2 工作流活动

- `request_broker_captcha`
- `submit_broker_login`
- `refresh_broker_session`
- `query_broker_order_status`

活动输入输出建议：

| 活动 | 输入 | 输出 |
| --- | --- | --- |
| `request_broker_captcha` | `account_id`、`broker_code` | `captcha_job_id`、`image_ref` |
| `submit_broker_login` | `account_id`、`credential_ref`、`captcha_text` | `broker_session_id`、`adapter_session_status` |
| `refresh_broker_session` | `broker_session_id` | `adapter_session_status`、`expires_at` |
| `query_broker_order_status` | `account_id`、`channel_order_id` | `raw_status`、`normalized_report` |

## 9. 实施要点

- 所有券商报文必须保留原始归档，所有状态映射必须集中维护。
- 券商登录失败原因必须结构化记录，供交易网关、前端和 CLI 返回可读提示。
- 必须提供首批券商能力矩阵样例，至少覆盖支持市场、盘前盘后、最小下单单位、撤单限制和可查询字段。
- 必须提供券商错误码到统一错误码的映射细则，未知错误统一落到保底错误族并保留原始错误引用。
