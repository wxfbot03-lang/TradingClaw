# TradingClaw 网关与平台基础详细设计

## 1. 范围说明

- 本文档覆盖 `api-gateway` 相关设计。
- 对应需求主要承接 `APP-001`，并为其余模块提供统一入口与公共治理能力。
- 本模块不承担具体业务事实存储，不实现交易、策略、行情、AI 的领域逻辑。

## 2. 模块职责

- 提供前端、CLI、外部客户端统一访问入口。
- 负责用户鉴权、限流、请求追踪、能力探测、审计入口。
- 统一响应信封、错误码映射、空字段清洗。
- 提供 WebSocket 订阅接入层，用于行情、订单、策略、通知推送。

## 3. 服务边界

### 3.1 本模块负责

- HTTP/JSON API 入口。
- WebSocket 连接管理与订阅鉴权。
- 请求级 `request_id`、`trace_id` 注入。
- 路由到内部领域服务。
- 能力开放版本校验与降级提示。

### 3.2 本模块不负责

- 用户身份的最终认证事实，认证事实由 `identity-service` 负责。
- 账户归属和交易权限事实，由 `account-service`、`audit-risk-service` 负责。
- 业务状态机推进，由对应领域服务负责。

## 4. 核心组件

| 组件 | 职责 |
| --- | --- |
| `auth-middleware` | 鉴权、会话校验、身份透传 |
| `rate-limit-middleware` | 用户级、IP 级、接口级限流 |
| `capability-filter` | 校验能力版本、开放状态和降级策略 |
| `response-normalizer` | 统一响应结构、错误码和空字段清洗 |
| `websocket-hub` | 连接管理、订阅鉴权、推送路由 |
| `audit-entry` | 关键操作审计入口记录 |

## 5. 核心流程

### 5.1 通用请求入口

1. 客户端请求进入 `api-gateway`。
2. 注入 `request_id`、`trace_id`。
3. 执行鉴权、限流、黑白名单、能力版本校验。
4. 路由至目标服务。
5. 统一映射响应信封并返回客户端。

### 5.2 命令类请求入口

适用于登录、下单、创建策略等写操作：

1. 接收客户端请求并完成参数校验。
2. 透传身份与上下文信息。
3. 对需要幂等的接口校验幂等键存在性。
4. 将请求转发到对应领域服务。
5. 返回受理结果或同步拒绝结果。

### 5.3 WebSocket 推送入口

1. 客户端发起握手。
2. 网关校验会话、订阅主题、能力开放状态。
3. 建立连接并按 `user_id/account_id/strategy_id` 建立路由。
4. 消费内部推送事件并投递给合法订阅者。

## 6. 接口设计

### 6.1 HTTP 资源建议

- `/api/v1/auth/*`
- `/api/v1/users/*`
- `/api/v1/accounts/*`
- `/api/v1/market/*`
- `/api/v1/news/*`
- `/api/v1/trades/*`
- `/api/v1/strategies/*`
- `/api/v1/ai/*`
- `/api/v1/capabilities/*`

### 6.1.1 通用请求头

| 请求头 | 必填 | 说明 |
| --- | --- | --- |
| `Authorization` | 按接口要求 | Bearer Token，会话类接口可为空 |
| `X-Request-Id` | 否 | 幂等追踪与链路追踪，未传则由网关生成 |
| `X-Idempotency-Key` | 命令接口必填 | 下单、撤单、创建策略等写接口使用 |
| `X-Client-Type` | 否 | `web`、`cli`、`service` 等 |
| `X-Client-Version` | 否 | 客户端版本，用于能力兼容与灰度判定 |

### 6.1.2 通用查询参数

| 参数 | 类型 | 说明 |
| --- | --- | --- |
| `page` | integer | 分页页码，默认 `1` |
| `page_size` | integer | 分页大小，默认 `20`，最大 `100` |
| `sort_by` | string | 排序字段 |
| `sort_order` | string | `asc` 或 `desc` |

### 6.2 响应信封

```json
{
  "request_id": "req_001",
  "code": "OK",
  "message": "success",
  "data": {},
  "meta": {
    "trace_id": "trc_001"
  }
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `request_id` | string | 请求唯一标识 |
| `code` | string | 统一错误码或成功码 |
| `message` | string | 可读信息 |
| `data` | object | 业务返回体 |
| `meta.trace_id` | string | 链路追踪标识 |
| `meta.pagination` | object | 分页接口可选返回 |
| `meta.capability_status` | object | 能力未开放时返回版本与降级信息 |

### 6.3 错误码入口规则

- 网关错误统一使用 `GW-*` 前缀。
- 下游领域错误在网关做稳定映射，不直接暴露内部栈信息。
- 未开放能力必须返回明确版本信息与降级说明。

### 6.4 WebSocket 消息结构

建议统一消息信封：

```json
{
  "event_type": "quote.updated",
  "event_version": 1,
  "request_id": "req_001",
  "trace_id": "trc_001",
  "data": {},
  "meta": {
    "channel": "market.quote"
  }
}
```

订阅请求建议字段：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `action` | string | 是 | `subscribe` 或 `unsubscribe` |
| `channel` | string | 是 | 如 `market.quote`、`trade.order` |
| `resource_id` | string | 否 | 如 `symbol`、`order_id`、`strategy_id` |
| `session_token` | string | 是 | 会话令牌 |

推送通道建议：

- `market.quote`
- `trade.order`
- `strategy.status`
- `notification.system`

## 7. 安全与治理

- 所有入口必须支持鉴权、签名校验、限流、审计入口。
- 运维和调试接口与业务接口隔离部署。
- WebSocket 订阅必须二次校验用户与资源关系。
- 网关不缓存最终业务事实，只缓存短期能力探测和限流令牌。

## 8. 可观测性

- 每个请求必须记录访问日志、trace、延迟、结果码。
- 对下游依赖记录服务级成功率、超时率、重试率。
- 对 WebSocket 记录连接数、订阅数、推送成功率。

## 9. 依赖与实施顺序

- 本模块应作为第一批落地。
- 与 `identity-service`、`capability-registry-service` 联调后即可支撑其余模块的统一入口。
- 后续所有模块的外部访问规范都以本模块为准。
