# TradingClaw 配置与密钥管理详细设计

## 1. 文档定位

- 本文档覆盖 TradingClaw 全平台的配置模型、配置分层、密钥管理、配置发布、启动时解析与配置治理规则。
- 本文档不描述具体业务字段含义，业务配置的语义仍由各领域文档维护；本文档定义这些配置应如何组织、存放、注入、校验和变更。
- 所有服务必须共享同一套配置与密钥治理规则，禁止每个服务单独发明自己的 `.env`、Yaml 层级、优先级和密钥注入方式。

## 2. 设计目标

- 建立跨服务一致、可审计、可复现、可回滚的配置体系。
- 明确“启动引导配置”“静态运行配置”“动态治理配置”“敏感密钥”“业务主数据”之间的边界。
- 保证服务在启动时可确定地装载配置，避免线上依赖不稳定的远程配置中心完成首启。
- 保证密钥从不以明文进入 Git 仓库、镜像层、长期日志或业务主库。
- 支持多环境、多区域、多租户扩展，但保持相同的配置契约和命名空间。

## 3. 配置总体原则

### 3.1 核心原则

- 配置必须声明式管理，默认通过 GitOps 管理非敏感配置，通过密钥管理系统管理敏感配置。
- 服务只读取“渲染后的运行快照”，不以远程配置服务作为同步请求链路上的强依赖。
- 所有配置都必须有 Schema、默认值策略、变更级别和 owner。
- 运行参数与业务事实严格分离；交易规则、能力开放状态、风险规则等若具备业务 owner，则属于业务数据或治理数据，不属于基础运行配置文件。
- 同一配置键只能有一个最终 owner；跨域引用不代表可随意覆盖。

### 3.2 配置对象分类

| 类型 | 典型内容 | 存储位置 | 变更频率 | 是否敏感 | owner |
| --- | --- | --- | --- | --- | --- |
| 启动引导配置 | `app_env`、`service_name`、`config_profile`、`region` | 环境变量 / Pod 注入 | 低 | 否 | 平台工程 |
| 静态运行配置 | HTTP 端口、线程池、Kafka topic、超时、重试 | Git 管理的配置包 | 中 | 否 | 平台工程 + 服务 owner |
| 基础设施端点配置 | MySQL host、Kafka brokers、Temporal namespace | Git 管理的配置包 | 低 | 否 | 平台工程 |
| 敏感密钥引用 | `vault://...`、证书路径、密钥版本 | Git 管理的引用配置 | 中 | 否（引用本身） | 平台工程 + 安全 |
| 敏感密钥实体 | DB 密码、JWT 私钥、券商密码、API Secret | Vault / KMS / CSI | 中 | 是 | 安全 / 对应业务 owner |
| 动态治理配置 | 能力矩阵、风控规则、模板开关 | MySQL / 管理服务 | 中高 | 视内容而定 | 对应治理域 |
| 业务主数据 | 账户、订单、策略配置快照 | MySQL | 高 | 视内容而定 | 对应业务域 |

边界约束：

- 风控规则、能力开放矩阵、策略模板 Schema 等若需要在线变更，应由对应业务服务管理，不应回写到基础运行配置文件。
- 市场标的、交易规则读模型、账户能力投影都属于业务事实或业务投影，不属于部署配置。

## 4. 配置分层模型

### 4.1 统一分层

运行时配置按以下层次装载：

1. 代码内安全默认值。
2. 平台共享基线配置 `shared/base.yaml`。
3. 环境级配置 `environments/{env}/global.yaml`。
4. 区域级覆盖 `environments/{env}/regions/{region}.yaml`。
5. 服务基线配置 `services/{service}/base.yaml`。
6. 服务环境覆盖 `services/{service}/{env}.yaml`。
7. 角色覆盖 `services/{service}/roles/{role}.yaml`。
8. 节点或实例临时覆盖 `runtime/overrides/*.yaml`，仅限运维紧急场景。
9. 启动环境变量，仅限引导参数和极少量显式覆盖。
10. CLI 参数，仅限本地调试或一次性作业。

### 4.2 覆盖优先级规则

- 后加载层覆盖先加载层，但不允许跨类型覆盖，例如密钥实体不可用普通配置文件直接替换。
- 生产环境默认禁止未知键、禁止空值覆盖、禁止使用 CLI 参数覆盖关键安全配置。
- `runtime/overrides` 只能由平台运维在紧急变更窗口内生成，并要求 TTL、变更单和自动清理。
- 同一个键若在多个文件重复定义，必须能通过渲染产物追踪最终来源。

### 4.4 渲染产物规则

- 每个可部署单元最终只消费一份渲染后的 `application.yaml` 和一份密钥挂载目录。
- 渲染产物必须包含 `config_version`、`rendered_at`、`source_commit`、`environment`、`region`、`service_name`、`role`。
- 渲染产物必须不可手工在线修改；任何变更都应回到配置源或密钥管理系统重新发布。

## 5. 配置命名空间与键规范

### 5.1 顶层命名空间

```yaml
app:
http:
grpc:
websocket:
database:
redis:
messaging:
temporal:
storage:
search:
timeseries:
security:
observability:
runtime:
integrations:
feature_flags:
```

### 5.2 命名规则

- 键名统一使用小写蛇形或小写点路径，不允许混用驼峰和随意缩写。
- 环境变量统一转换为大写下划线，并以 `TC_` 前缀开头。
- 布尔配置必须使用肯定语义，例如 `enabled`、`strict_validation`，避免双重否定。
- 时间值必须显式单位化，统一使用 `_seconds`、`_milliseconds`、`_minutes`、`_ttl_seconds` 后缀。
- 容量值必须显式单位化，统一使用 `_bytes`、`_mb`、`_count`。

### 5.3 环境变量分组

| 前缀 | 说明 |
| --- | --- |
| `TC_BOOTSTRAP_*` | 启动引导配置 |
| `TC_APP_*` | 服务基础标识 |
| `TC_HTTP_*` / `TC_GRPC_*` | 接入监听参数 |
| `TC_DB_*` | MySQL / 事务库 |
| `TC_REDIS_*` | Redis |
| `TC_KAFKA_*` | Kafka / Schema Registry |
| `TC_TEMPORAL_*` | Temporal |
| `TC_S3_*` / `TC_STORAGE_*` | 对象存储 |
| `TC_OTEL_*` / `TC_LOG_*` | 可观测性 |
| `TC_VAULT_*` | 密钥引导参数 |
| `TC_OVERRIDE_*` | 临时覆盖与运维紧急参数 |

### 5.4 必选引导环境变量

| 变量 | 说明 |
| --- | --- |
| `TC_BOOTSTRAP_ENV` | `local`、`dev-integration`、`qa`、`staging`、`prod-primary`、`prod-dr` |
| `TC_BOOTSTRAP_REGION` | 区域或可用区标识 |
| `TC_BOOTSTRAP_SERVICE` | 服务名 |
| `TC_BOOTSTRAP_ROLE` | 进程角色 |
| `TC_BOOTSTRAP_CONFIG_PATH` | 渲染后配置文件路径 |
| `TC_BOOTSTRAP_SECRETS_PATH` | 密钥挂载目录 |
| `TC_BOOTSTRAP_CONFIG_VERSION` | 配置版本号 |

## 6. 密钥管理设计

### 6.1 统一方案

- 生产、预发、测试环境统一采用 Vault 作为密钥管理系统，根密钥由云 KMS 或 HSM 托管。
- Kubernetes 中通过 External Secrets Operator + CSI Secret Store 把密钥投递为只读文件或短期环境变量。
- 本地开发环境仍使用同一套密钥模型，可接入开发 Vault 命名空间，不允许把生产或测试密钥导出到个人 `.env`。

### 6.2 密钥存储分层

| 层级 | 内容 | 存储位置 |
| --- | --- | --- |
| 平台级密钥 | JWT 签名密钥、内部 mTLS 证书、Kafka SASL 凭据 | Vault 平台路径 |
| 服务级密钥 | MySQL 用户密码、Redis 密码、对象存储密钥 | Vault 服务路径 |
| 通道级密钥 | 券商凭据、交易所 API Secret、Webhook Token | Vault 业务路径 |
| 轮换材料 | 新旧密钥版本、证书链、过渡期双活密钥 | Vault 版本化路径 |

推荐路径：

```text
kv/tradingclaw/{environment}/{service}/...
kv/tradingclaw/{environment}/platform/...
kv/tradingclaw/{environment}/accounts/{account_id}/...
pki/tradingclaw/{environment}/...
```

### 6.3 密钥引用格式

统一采用 URI 风格引用：

```text
vault://kv/tradingclaw/prod-primary/trade-gateway-service#mysql_password
vault://kv/tradingclaw/prod-primary/platform#jwt_private_key
file:///var/run/secrets/tradingclaw/tls/server.crt
```

规则：

- 配置文件中只允许保存引用，不允许保存明文。
- 应用侧统一通过密钥解析器装载引用，业务代码不得自行解析 Vault 协议。
- 业务库中仅保存 `credential_ref`、`secret_path`、`credential_version` 等元数据，不保存真实密钥实体。

### 6.4 密钥轮换策略

- 数据库、Redis、Kafka、对象存储等基础设施凭据必须支持季度轮换。
- JWT、Cookie 加密密钥、内部证书必须支持双版本并行窗口。
- 券商和交易所接入密钥轮换必须保留版本号和失效时间，并支持引用级别切换而非明文覆盖。
- 轮换过程必须支持“新旧并行 -> 灰度切换 -> 清理旧版本”的三阶段模型。

### 6.5 严禁事项

- 严禁把明文密钥提交到 Git、镜像、CI 日志、工单系统或聊天工具。
- 严禁服务在异常日志中打印密钥、Token、Cookie、验证码识别结果或完整连接串。
- 严禁把用户券商密码、交易所 Secret 或 MFA 材料落入 MySQL 业务表。

## 7. 服务配置契约

### 7.1 每个服务必须具备的配置块

| 配置块 | 必需字段 |
| --- | --- |
| `app` | `service_name`、`role`、`env`、`region`、`version` |
| `http` | `enabled`、`host`、`port`、`workers`、`request_timeout_seconds` |
| `grpc` | `enabled`、`host`、`port`、`max_receive_mb` |
| `database` | `dsn_ref`、`pool_size`、`pool_recycle_seconds` |
| `redis` | `mode`、`address_ref`、`db` 或 `cluster_nodes_ref` |
| `messaging.kafka` | `brokers_ref`、`client_id`、`schema_registry_ref` |
| `temporal` | `namespace`、`endpoint_ref`、`task_queue` |
| `security` | `jwt_public_key_ref`、`tls_mode`、`cors_policy` |
| `observability` | `log_level`、`otel_endpoint_ref`、`metrics_enabled` |
| `runtime` | `shutdown_grace_seconds`、`strict_validation`、`readiness_timeout_seconds` |

### 7.2 角色附加配置

| 角色 | 额外配置 |
| --- | --- |
| `http-api` | 路由前缀、反向代理信任配置、限流参数 |
| `websocket` | Redis 路由前缀、最大连接数、订阅配额 |
| `grpc-api` | gRPC 拦截器、并发上限、反压参数 |
| `event-consumer` | 消费组、topic 列表、批量大小、回退策略 |
| `temporal-worker` | `task_queue`、并发执行数、活动超时 |
| `scheduler` | 定时计划、互斥锁配置、漂移补偿 |
| `adapter-worker` | egress 策略、浏览器会话或签名上下文配置 |
| `migration-job` | 目标 schema、迁移锁、执行窗口 |

### 7.3 示例渲染配置

```yaml
app:
  service_name: trade-gateway-service
  role: http-api
  env: prod-primary
  region: cn-hz-a
  version: 1.0.0
  config_version: cfg-20260318-001

http:
  enabled: true
  host: 0.0.0.0
  port: 8080
  workers: 4
  request_timeout_seconds: 15

database:
  dsn_ref: vault://kv/tradingclaw/prod-primary/trade-gateway-service#mysql_dsn
  pool_size: 30
  pool_recycle_seconds: 1800

messaging:
  kafka:
    brokers_ref: vault://kv/tradingclaw/prod-primary/platform#kafka_brokers
    schema_registry_ref: vault://kv/tradingclaw/prod-primary/platform#schema_registry_url
    client_id: trade-gateway-service

runtime:
  strict_validation: true
  readiness_timeout_seconds: 60
  shutdown_grace_seconds: 45
```

## 8. 配置发布与变更流程

### 8.1 配置发布链路

1. 平台或服务 owner 修改配置源文件。
2. CI 进行 Schema 校验、策略校验、引用完整性校验和差异审查。
3. 渲染生成环境级运行配置快照。
4. 非敏感配置打包为 ConfigMap 或只读文件，敏感配置通过 Vault 引用和 SecretStore 绑定。
5. Argo CD 或等价 GitOps 控制器把新配置发布到目标环境。
6. 服务通过滚动更新或显式重载消费新配置。

### 8.2 变更级别

| 级别 | 示例 | 要求 |
| --- | --- | --- |
| L1 | 日志级别、缓存 TTL、限流阈值 | 双人审核，可滚动发布 |
| L2 | Kafka topic、线程池、外部端点切换 | 双人审核 + 灰度验证 |
| L3 | 密钥引用切换、认证证书、网络白名单 | 安全审核 + 变更窗口 |
| L4 | 环境拓扑、数据库主从、区域切换 | 架构评审 + 发布演练 |

### 8.3 动态配置重载规则

- 基础运行配置默认通过重新部署生效，避免“部分实例已重载、部分未重载”的不确定状态。
- 仅以下配置允许热更新：日志级别、部分限流阈值、只读超时参数、非安全级别客户端熔断参数。
- 数据源、证书、监听端口、Schema 严格模式、消息 topic 映射等关键配置不得热更新，必须通过重新发布生效。

## 9. 启动时配置装载与校验

### 9.1 装载顺序

1. 读取引导环境变量。
2. 定位渲染后的 `application.yaml`。
3. 解析密钥引用并装载密钥实体。
4. 以 Pydantic Settings + JSON Schema 执行强校验。
5. 执行跨字段校验，例如启用 Kafka 时必须有 Schema Registry，引入 WebSocket 时必须配置 Redis 路由。
6. 生成不可变运行快照，交给应用容器使用。

### 9.2 校验失败策略

- 生产环境配置校验失败必须启动失败，禁止降级为默认值继续运行。
- 测试和本地环境允许显示缺省提示，但默认仍建议 fail fast。
- 任一密钥解析失败、引用不存在、版本不兼容或未知键出现，都应视为启动阻断项。

## 10. 配置治理与审计

### 10.1 owner 规则

- 平台工程拥有共享命名空间、配置结构、渲染器和发布流程。
- 每个服务 owner 拥有自己服务配置块的字段定义和默认值策略。
- 安全团队拥有密钥路径规划、轮换窗口和审计规则。

### 10.2 审计要求

- 每次配置变更都必须记录提交人、审批人、差异摘要、目标环境和发布时间。
- 每次密钥轮换都必须记录旧版本、新版本、生效时间、失效时间和回滚窗口。
- 服务启动日志必须记录 `config_version`、`source_commit`、`env`、`region`，但不得记录密钥明文。

## 11. 禁止反模式

- 每个服务自带一套互不兼容的 `.env.example` 与启动规则。
- 把基础设施地址写死在代码中。
- 让应用在运行中直接依赖远程配置中心完成强一致拉取。
- 通过 Helm values、环境变量和代码默认值三处同时定义同一关键配置而无单一真源。
- 把业务开关、策略参数、风险规则、能力矩阵混进基础设施配置文件。

## 12. 结论

TradingClaw 的配置体系必须是“Git 管理非敏感配置 + Vault 管理敏感密钥 + 渲染后快照启动 + 严格 Schema 校验 + 明确 owner 和审计链路”的统一模型。后续所有服务、脚本、容器、作业和运维操作都必须围绕这套模型实现，而不是各自维护局部约定。
