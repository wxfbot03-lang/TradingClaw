## 2. 模块职责

### 2.1 `ai-orchestration-service`

- 问答编排、模型配置读取与写入。
- 工具调用、上下文聚合、结果治理。

### 2.2 `ocr-service`

- 验证码识别与结果标准化。
- OCR 任务记录、结果追踪、错误返回。

### 2.3 `capability-registry-service`

- 跨端能力版本矩阵维护。
- 能力开放状态、降级策略、工具可见性控制。

## 3. 领域边界

- AI 只负责编排与增强，不直接持有交易主链路状态。
- OCR 只负责识别，不负责券商登录主状态机。
- 能力注册中心只维护能力元数据，不维护实际业务状态。

## 5. 核心业务流程

### 5.1 AI 问答与工具调用

1. 接收用户问题。
2. 查询 `capability-registry-service` 获取允许使用的工具和版本。
3. 聚合行情、资讯、策略、交易上下文。
4. 调用模型并执行工具链。
5. 输出结果并发布工具调用审计事件。

治理约束：

- 必须明确首批模型供应商、超时阈值、token 限额、单次工具调用预算和失败回退策略。
- 涉及交易、账户、OCR 的上下文在入模前必须执行脱敏和最小必要字段裁剪。
- AI 结果可解释但不可越权执行交易写操作。

### 5.2 OCR 验证码识别

1. 接收图片或任务请求。
2. 调用 OCR 供应商识别。
3. 返回标准化文本、错误码、置信度。
4. 发布 `ocr.job_finished` 事件。

治理约束：

- 必须明确首批 OCR 供应商、主备 fallback、单次超时和重试上限。
- 验证码图像和识别文本必须按敏感信息策略设置保留期和脱敏边界。

### 5.3 能力版本治理

1. 维护能力 ID、开放版本、跨端可见性、降级策略。
2. 为网关和 AI 工具调用提供能力判断接口。
3. 对未开放能力返回统一降级说明。

## 6. 数据设计

核心表：

- `model_configs`
- `ai_conversations`
- `ai_tool_calls`
- `ocr_jobs`
- `capability_definitions`
- `capability_versions`

设计要点：

- 模型配置、AI 会话、工具调用记录、OCR 任务、能力定义与版本矩阵建议落 MySQL，便于审计和版本回溯。
- 大模型上下文长文本、OCR 原图、工具调用原始输入输出建议以对象存储保存，MySQL 只保留索引和摘要。
- Redis 可用于短期会话上下文缓存、能力查询热点缓存和工具调用去重键，但不能替代会话与审计主记录。
- 默认值建议：`ai_conversations.status = ACTIVE`，`ai_tool_calls.status = PENDING`，`ocr_jobs.status = CREATED`，`capability_versions.enabled = false`。
- 空值规则：`response_ref`、`text_ref`、`degrade_strategy` 可按场景为空；`conversation_id`、`tool_name`、`job_id`、`capability_id`、`open_version` 默认不可空。
- 删除策略：AI 会话和工具调用默认只做归档或脱敏，不直接物理删除；能力版本通过 `enabled` 和版本切换管理下线。
- 审计要求：模型配置、能力版本、工具调用记录建议补 `created_by`、`updated_by`、`provider_code`，便于回溯供应商和操作者来源。
- `enabled` 固定表示是否启用；开放态、降级态等扩展语义统一放在平台治理字段 `capability_status` 中，避免与账户域 `account_capability_status` 混用。
- MVP 阶段允许 `ai-orchestration-service`、`ocr-service`、`capability-registry-service` 逻辑分层、物理合并部署，稳定后再独立扩展。

## 8. 接口设计

### 8.1 HTTP 入口

- `/api/v1/ai/chat`
- `/api/v1/ai/models`
- `/api/v1/ocr/jobs`
- `/api/v1/ocr/jobs/{id}`
- `/api/v1/capabilities`

#### 8.1.1 `POST /api/v1/ai/chat`

必需请求头：`Authorization`

请求体：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `conversation_id` | string | 否 | 不传则创建新会话 |
| `question` | string | 是 | 用户问题 |
| `context_scope` | array | 否 | 如 `market`、`news`、`strategy`、`trade` |
| `tool_preference` | array | 否 | 允许优先使用的工具 |

返回体 `data`：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `conversation_id` | string | 会话 ID |
| `answer` | string | 最终回答 |
| `tool_calls` | array | 工具调用摘要 |
| `model_name` | string | 使用模型 |

语义约束：

- AI 聊天属于同步返回结果的增强接口，完成态以 `conversation_id` 为主键观测，不额外返回 `workflow_id`。

#### 8.1.2 `GET /api/v1/ai/models`

返回体 `data`：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `items` | array | 模型列表 |
| `items[].model_id` | string | 模型 ID |
| `items[].provider_code` | string | 提供方 |
| `items[].enabled` | boolean | 是否启用 |

#### 8.1.3 `POST /api/v1/ocr/jobs`

必需请求头：`Authorization`、`X-Idempotency-Key`

请求体：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `image_ref` | string | 是 | 图片引用或上传返回 ID |
| `scene` | string | 是 | 如 `broker_captcha` |

返回体 `data`：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `job_id` | string | OCR 任务 ID |
| `status` | string | 任务状态 |

语义约束：

- 该接口为任务型资源创建接口，创建时只保证任务已登记。
- 最终识别结果通过查询接口获取，不在创建响应中混放最终结果字段。

#### 8.1.4 `GET /api/v1/ocr/jobs/{id}`

路径参数：`id` 为 OCR 任务 ID。

返回体 `data`：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `job_id` | string | OCR 任务 ID |
| `status` | string | 任务状态 |
| `text` | string | 识别结果，可空 |
| `confidence` | number | 置信度，可空 |
| `reason_code` | string | 失败原因码，可空 |

#### 8.1.5 `GET /api/v1/capabilities`

查询参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `client_type` | string | 否 | `web`、`cli`、`ai` |
| `version` | string | 否 | 客户端版本 |

返回体 `data`：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `items` | array | 能力列表 |
| `items[].capability_id` | string | 能力 ID |
| `items[].enabled` | boolean | 是否启用 |
| `items[].capability_status` | string | 能力开放状态 |
| `items[].open_version` | string | 开放版本 |
| `items[].degrade_strategy` | string | 降级策略 |

## 9. 实施要点

- AI 编排接口建议使用 FastAPI + Pydantic 实现，请求上下文、工具调用结果和能力配置统一走 Python 数据模型收口。
- 工具调用工作流建议复用 Temporal Python SDK，避免 AI 编排、OCR 任务、能力治理各自维护一套后台任务框架。
- AI 工具调用必须经过工具白名单和资源权限校验。
- AI 可以提供建议和解释，但不能绕过交易风控直接修改交易事实。
- 能力注册中心必须成为前端、CLI、AI 的统一版本口径来源。
