# TradingClaw 智能能力与治理详细设计

## 1. 文档定位

- 本文档覆盖 `ai-orchestration-service`、`ocr-service`、`capability-registry-service`，定义 AI 编排、基于大语言模型的 OCR 识别和平台能力治理的统一设计。
- 本模块为问答增强、工具调用、验证码识别和跨端能力开放控制提供基础支撑。
- AI 和 OCR 只提供辅助能力与编排能力，不直接持有交易主链路状态，也不能越权执行真实交易写操作。

## 2. 模块定位

### 2.1 `ai-orchestration-service`

- 负责问答编排、LiteLLM 路由配置读取、工具调用和上下文聚合。
- 负责将行情、资讯、策略和交易等外部上下文组织为可控的 AI 输入。

### 2.2 `ocr-service`

- 负责基于大语言模型的验证码识别和 OCR 任务标准化结果输出。
- 负责 OCR 任务记录、结果追踪和错误返回。

### 2.3 `capability-registry-service`

- 负责跨端能力版本矩阵、开放状态、降级策略和工具可见性控制。
- 负责为网关、前端、CLI 和 AI 提供统一能力开放口径。

## 3. 领域边界

- AI 只负责编排、建议和增强，不直接持有交易订单、策略状态或账户事实。
- OCR 只负责识别，不负责券商登录或交易认证主状态机。
- 能力注册中心只维护能力元数据和开放治理，不维护业务主状态。
- 平台能力开放状态统一使用 `capability_status`，不得与账户域 `account_capability_status` 混用。
- 大语言模型统一通过外部 LiteLLM 服务接入；业务域不直接对接具体模型供应商 API。

## 5. 智能能力主线

### 5.1 AI 问答与工具调用主线

流程如下：

1. 接收用户问题。
2. 查询 `capability-registry-service`，获取当前客户端和版本允许使用的工具与能力。
3. 聚合行情、资讯、策略、交易等上下文。
4. 在入模前执行脱敏、权限裁剪和最小必要字段过滤。
5. 调用外部 LiteLLM 服务并执行工具链。
6. 输出结果，并发布工具调用审计事件。

治理约束：

 - 首批模型接入以外部 LiteLLM 服务为统一入口，需明确模型路由、超时阈值、token 限额、单次工具调用预算和失败回退策略。
 - 涉及交易、账户和 OCR 的上下文在入模前必须执行脱敏和最小必要字段裁剪。
 - AI 结果可解释，但不可越权执行交易写操作。

### 5.2 OCR 识别主线

流程如下：

1. 接收图片引用或 OCR 任务请求。
2. 调用基于大语言模型的视觉识别能力执行 OCR；MVP 阶段统一经 LiteLLM 路由到外部模型服务。
3. 返回标准化文本、错误码和置信度。
4. 发布 `ocr.job_finished` 事件。

治理约束：

 - OCR 采用大语言模型识别时，必须明确主模型、备用模型、单次超时和重试上限。
 - 验证码图像和识别文本必须按敏感信息策略设置保留期和脱敏边界。

### 5.3 平台能力治理主线

流程如下：

1. 维护能力 ID、开放版本、跨端可见性和降级策略。
2. 为网关和 AI 工具调用提供能力判断接口。
3. 对未开放能力返回统一降级说明。

治理约束：

- `enabled` 固定表示配置是否启用；开放态、降级态等扩展语义统一使用 `capability_status`。
- 能力注册中心必须成为前端、CLI、AI 和网关的统一版本口径来源。

## 6. 数据设计

核心表：

- `model_configs`
- `ai_conversations`
- `ai_tool_calls`
- `ocr_jobs`
- `capability_definitions`
- `capability_versions`

设计要点：

- LiteLLM 路由配置、AI 会话、工具调用记录、OCR 任务、能力定义与版本矩阵建议落 MySQL，便于审计和版本回溯。
- 大模型上下文长文本、OCR 原图和工具调用原始输入输出建议存对象存储，MySQL 只保留索引和摘要。
- Redis 可用于短期会话上下文缓存、能力查询热点缓存和工具调用去重键，但不能替代会话与审计主记录。
- 默认值建议：`ai_conversations.status = ACTIVE`，`ai_tool_calls.status = PENDING`，`ocr_jobs.status = CREATED`，`capability_versions.enabled = false`。
- AI 会话和工具调用默认只做归档或脱敏，不直接物理删除；能力版本通过 `enabled` 和版本切换管理下线。
- LiteLLM 路由配置、能力版本和工具调用记录建议补 `created_by`、`updated_by`、`provider_code`，便于回溯上游模型提供方和操作者来源。
- MVP 阶段允许 `ai-orchestration-service`、`ocr-service`、`capability-registry-service` 逻辑分层、物理合并部署，稳定后再独立扩展。

## 7. 事件设计

核心事件：

- `conversation.created`
- `tool_call.executed`
- `ocr.job_finished`

事件约束：

- 工具调用事件应保留资源引用与结果状态，供审计和追踪使用。
- OCR 事件只表达识别任务完成结果，不表达业务认证主流程状态。

## 9. 实施要点

- AI 编排接口建议使用 FastAPI + Pydantic 实现，请求上下文、工具调用结果和能力配置统一通过 Python 数据模型收口。
- 工具调用工作流建议复用 Temporal Python SDK，避免 AI 编排、OCR 任务和能力治理各自维护独立后台任务框架。
- AI 工具调用必须经过工具白名单、资源权限校验和能力开放判断。
- AI 可以提供建议和解释，但不能绕过交易风控直接修改交易事实。

## 10. LiteLLM 接口设计

当前设计前提是：大语言模型统一通过外部 LiteLLM 服务接入，OCR 也通过大语言模型能力完成，不再单独依赖传统 OCR 厂商接口。

LiteLLM 作为外部模型网关时，本系统优先依赖其 OpenAI 兼容接口和 OCR 接口，不需要外部再额外为我们定制 API。

### 10.1 接入约束

- `ai-orchestration-service` 优先通过 LiteLLM Proxy 的 OpenAI 兼容接口访问模型。
- 问答与工具调用主链路优先使用 `/v1/chat/completions`。
- 如后续需要统一响应式接口，可补充接入 `/v1/responses`，但 MVP 可不强依赖。
- OCR 场景优先使用 LiteLLM 的 `/v1/ocr` 接口。
- 认证方式默认使用 `Authorization: Bearer <token>`。

### 10.2 LiteLLM 对话接口

推荐使用：`POST /v1/chat/completions`

用途：

- AI 问答
- 工具调用
- 多模型路由
- 流式回答

建议请求体字段：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `model` | string | 是 | LiteLLM 中配置的模型名或路由名 |
| `messages` | array | 是 | OpenAI 兼容消息列表 |
| `tools` | array | 否 | 可调用工具定义 |
| `tool_choice` | string/object | 否 | 工具调用策略 |
| `stream` | boolean | 否 | 是否流式返回 |
| `temperature` | number | 否 | 采样温度 |
| `max_tokens` | integer | 否 | 最大生成 token |
| `metadata` | object | 否 | 可附带 `user_id`、`conversation_id`、`trace_id` 等上下文 |

建议响应关注字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | string | 调用结果 ID |
| `model` | string | 实际使用模型 |
| `choices[].message.content` | string | 模型回答 |
| `choices[].message.tool_calls` | array | 工具调用结果 |
| `usage.prompt_tokens` | integer | 输入 token 数 |
| `usage.completion_tokens` | integer | 输出 token 数 |
| `usage.total_tokens` | integer | 总 token 数 |

补充说明：

- 该接口满足当前 AI 问答主线和工具调用主线的主要需求。
- 如果需要流式输出，沿用同一接口并设置 `stream=true`。

### 10.3 LiteLLM OCR 接口

推荐使用：`POST /v1/ocr`

用途：

- 验证码识别
- 图片文本提取
- 文档 OCR

建议请求体字段：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `model` | string | 是 | OCR 模型名，如 LiteLLM 中配置的视觉/OCR 模型 |
| `document` | object | 是 | 待识别对象 |
| `document.type` | string | 是 | `document_url`、`image_url` 或 `file` |
| `document.document_url` | string | 条件必填 | 文档 URL 或 base64 data URI |
| `document.image_url` | string | 条件必填 | 图片 URL 或 base64 data URI |
| `document.file` | string/bytes/file | 条件必填 | 文件路径、文件对象或原始字节 |
| `document.mime_type` | string | 否 | 文件 MIME 类型 |
| `pages` | array | 否 | 指定处理页码 |
| `include_image_base64` | boolean | 否 | 是否返回提取图片 |

建议响应关注字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `pages` | array | OCR 结果页列表 |
| `pages[].index` | integer | 页码 |
| `pages[].markdown` | string | 识别出的文本内容 |
| `model` | string | 实际使用模型 |
| `usage_info` | object | 处理统计信息 |
| `object` | string | 固定为 `ocr` |

补充说明：

- 当前 OCR 任务主要是验证码识别和轻量图片识别，MVP 可以只关注单图片输入。
- LiteLLM 的 OCR 接口遵循其支持的 OCR 提供方格式，适配时我们只抽取业务真正需要的结果字段。

### 10.4 建议接入的辅助接口

| 接口 | 用途 | 是否 MVP 必需 | 说明 |
| --- | --- | --- | --- |
| `GET /models` 或等价模型列表接口 | 获取可用模型清单 | 否 | 用于后台治理和配置校验 |
| `POST /v1/responses` | 统一响应式生成接口 | 否 | 后续如要统一多模态或 Responses API 再接入 |
| 健康检查接口 | 服务存活与可用性探测 | 建议 | 用于网关和运维探活 |

### 10.5 实施约束

- 业务代码不直接耦合某家模型厂商 SDK，统一通过 LiteLLM 接入。
- LiteLLM 返回的真实模型名和上游提供方信息应尽量记录到 `provider_code`、`model_id` 或审计日志中。
- OCR 的原图与识别结果仍按现有敏感信息规则落对象存储和摘要索引，不因改用 LiteLLM 改变保留边界。
