# TradingClaw 部署与发布详细设计

## 1. 文档定位

- 本文档覆盖 TradingClaw 在开发、测试、预发、生产和灾备环境中的目标部署架构、发布流水线、回滚策略、灾备方案、安全边界和运维治理要求。
- 本文档定义长期有效的目标态，不以短期开发周期妥协；局部实施阶段可以分批落地，但目标架构必须以本文档为准。
- 本文档与 `配置与密钥管理详细设计.md`、`运行启动与环境编排详细设计.md` 一起构成平台工程交付基线。

## 2. 部署目标

- 建立适用于长期演进的统一云原生部署体系，而不是一次性脚本式交付。
- 支持多环境隔离、多可用区高可用、跨区域灾备、灰度发布、审计留痕和密钥零明文分发。
- 保证交易主链路、策略运行时、审计风控和通知链路在基础设施层面具备明确的故障域隔离与恢复路径。
- 让部署方式本身成为平台能力，而不是由每个业务服务分别定义。

## 3. 目标技术冻结

### 3.1 统一平台选型

| 领域 | 冻结方案 |
| --- | --- |
| 容器编排 | Kubernetes |
| 部署控制 | Helm + Argo CD |
| 灰度发布 | Argo Rollouts |
| 服务网格 | Istio |
| 密钥管理 | Vault + External Secrets Operator + CSI |
| 事务库 | MySQL 8 高可用集群 |
| 缓存 | Redis Cluster |
| 消息总线 | Kafka |
| Schema 管理 | Confluent Schema Registry 或兼容实现 |
| 工作流引擎 | Temporal Cluster |
| 时序分析 | ClickHouse |
| 搜索 / 日志检索 | OpenSearch |
| 对象存储 | S3 兼容对象存储 |
| 指标 / 告警 | Prometheus + Alertmanager |
| 链路追踪 | OpenTelemetry Collector + Tempo |
| 日志采集 | Fluent Bit / Vector |
| 镜像仓库 | 企业级 OCI Registry |
| 制品签名 | Cosign |

### 3.2 设计解释

- Kubernetes 是唯一正式运行平面；Docker Compose 仅用于开发辅助，不作为正式部署基线。
- Istio 用于服务间 mTLS、流量治理、可观测性挂钩和金丝雀发布流量控制。
- Argo CD 保证环境状态声明式同步；Argo Rollouts 负责发布策略，而非手工 `kubectl set image`。

## 4. 环境与账户分层

### 4.1 目标环境矩阵

| 环境 | 目标 |
| --- | --- |
| `local` | 开发者本地与单机一致性验证 |
| `dev-integration` | 开发联调与共享依赖验证 |
| `qa` | 测试与回归验证 |
| `staging` | 预发验证、性能演练、灰度演练 |
| `prod-primary` | 生产主区域 |
| `prod-dr` | 生产灾备区域 |

### 4.2 隔离原则

- `prod-primary` 和 `prod-dr` 必须位于独立云账户或独立资源组。
- `staging` 与 `prod` 不共享任何数据库、Kafka、Vault、对象存储命名空间或证书根。
- 非生产环境可共享统一观测平台，但必须通过租户、索引或项目维度隔离。

## 5. 总体部署拓扑

### 5.1 逻辑分区

系统划分为五个基础区域：

1. 边缘接入区：CDN、WAF、API Ingress Gateway、外部负载均衡。
2. 应用服务区：业务服务 Pod、WebSocket、gRPC 服务、消费者、worker。
3. 平台服务区：Kafka、Schema Registry、Temporal、Vault、GitOps、Service Mesh 控制面。
4. 数据服务区：MySQL、Redis、ClickHouse、OpenSearch、对象存储。
5. 受限适配区：证券与数字资产适配服务、受控出网代理、第三方白名单 egress。

### 5.2 区域边界

- 边缘接入区只暴露 `api-gateway` 的入口，不直接暴露内部领域服务。
- 适配服务区需单独网络策略和节点池，避免第三方通道不稳定性影响核心业务服务。
- 平台服务区与应用服务区通过服务网格和网络策略限制访问范围。

## 6. Kubernetes 集群设计

### 6.1 集群策略

- `dev-integration`、`qa`、`staging` 各自拥有独立 Kubernetes 集群。
- `prod-primary` 和 `prod-dr` 分别拥有独立生产集群。
- 每个生产集群跨三个可用区部署，所有无状态服务至少分散到两个以上可用区。

### 6.2 命名空间规划

| Namespace | 说明 |
| --- | --- |
| `tc-edge` | Ingress Gateway、边缘接入组件 |
| `tc-core` | 身份、账户、市场、交易、策略等核心业务服务 |
| `tc-adapters` | 券商、交易所适配服务 |
| `tc-governance` | 风控、审计、通知、能力治理 |
| `tc-ai` | AI 编排与 OCR |
| `tc-platform` | GitOps、Vault Agent、Temporal、Schema Registry 等 |
| `tc-observability` | Prometheus、Tempo、日志采集与告警 |
| `tc-batch` | 回补、迁移、后台批处理 |

### 6.3 节点池规划

| 节点池 | 负载 |
| --- | --- |
| `edge-pool` | Ingress Gateway、WAF Agent |
| `core-api-pool` | HTTP / gRPC 核心服务 |
| `async-worker-pool` | Kafka consumer、Temporal worker、scheduler |
| `adapter-isolated-pool` | 证券/数字资产适配服务 |
| `batch-pool` | backfill、migration、admin-job |
| `observability-pool` | 观测组件 |

约束：

- `adapter-isolated-pool` 必须限制出网域名、IP 段和凭据访问路径。
- `core-api-pool` 和 `async-worker-pool` 必须设置不同的资源配额和伸缩策略。

## 7. 服务部署单元设计

### 7.1 部署单元规则

- 每个角色对应一个独立 Deployment、StatefulSet 或 Job。
- HTTP 与 WebSocket、API 与 Consumer、Worker 与 Scheduler 默认独立部署。
- Kafka consumer 和 Temporal worker 不与 `http-api` 同 Pod 运行。

### 7.2 推荐部署单元

| 服务 | 部署单元 |
| --- | --- |
| `api-gateway` | `api-gateway-http`、`api-gateway-websocket` |
| `identity-service` | `identity-http`、`identity-grpc`、`identity-consumer` |
| `account-service` | `account-http`、`account-grpc`、`account-consumer` |
| `market-service` | `market-http`、`market-grpc`、`market-backfill`、`market-scheduler` |
| `trade-gateway-service` | `trade-http`、`trade-grpc`、`trade-consumer`、`trade-temporal-worker` |
| `securities-trading-service` | `securities-grpc`、`securities-adapter-worker` |
| `crypto-trading-service` | `crypto-grpc`、`crypto-adapter-worker`、`crypto-scheduler` |
| `strategy-runtime-service` | `strategy-runtime-consumer`、`strategy-runtime-worker`、`strategy-runtime-scheduler` |
| `notification-service` | `notification-consumer`、`notification-scheduler` |

## 8. 流量入口与网络设计

### 8.1 外部入口链路

```text
Client -> CDN -> WAF -> External Load Balancer -> Istio Ingress Gateway -> api-gateway -> Internal Services
```

规则：

- 所有外部 HTTP / WebSocket 请求都必须通过统一边缘入口。
- gRPC 服务默认只在集群内部暴露，通过网格服务发现访问。
- 运维和调试接口与业务接口必须隔离，优先使用内网入口或零信任访问网关。

### 8.2 东西向通信

- 服务间通信默认通过 Istio mTLS。
- 交易、风控、账户、策略等核心服务必须启用严格网络策略，只开放必要端口和来源 namespace。
- 适配服务所有外部访问必须走 Egress Gateway 或受控 NAT，不允许业务 Pod 自由直连公网。

## 9. 数据平面设计

### 9.1 MySQL

- 业务数据库采用高可用 MySQL 集群，主库单写、跨可用区同步复制、只读副本分担查询。
- 每个业务域拥有独立 schema；严禁跨服务共享表。
- 开启 PITR、定期全量备份和 binlog 归档。

### 9.2 Redis

- 统一采用 Redis Cluster，用于热点缓存、幂等键、分布式锁、短期会话索引和限流。
- Redis 不承担最终事实；所有写入都必须可由 MySQL 或上游事实恢复。

### 9.3 Kafka 与 Schema Registry

- Kafka 集群跨三个可用区部署，topic 副本数不少于 3。
- Schema Registry 高可用部署，所有事件 Schema 采用兼容性策略管理。
- 生产环境禁止未注册 Schema 的自由 JSON 事件进入总线。

### 9.4 Temporal

- Temporal Cluster 独立部署在 `tc-platform`，使用专用持久化库和可视化存储。
- 业务服务只访问 Temporal Frontend，不直接依赖内部组件。
- Workflow namespace 按环境和业务域分层，例如 `prod-primary-trade`、`prod-primary-strategy`、`prod-dr-trade`。

### 9.5 ClickHouse、OpenSearch、对象存储

- 行情时序、指标和研究分析进入 ClickHouse。
- 资讯检索、业务日志检索和审计检索进入 OpenSearch。
- OCR 原图、原始报文、长文本上下文、审计附件与备份归档进入 S3 兼容对象存储。

## 10. 配置与密钥下发

### 10.1 下发模型

- 非敏感配置通过 Helm 渲染为 ConfigMap 或只读卷文件。
- 敏感密钥通过 External Secrets Operator 同步或以 CSI 只读卷方式挂载。
- Pod 启动时只拿到渲染后的配置快照和最小必要密钥，不直接获得 Vault 广域读权限。

### 10.2 密钥权限边界

- 服务账号只允许访问对应服务路径和共享平台路径。
- 通道适配服务若需要访问账户级密钥，必须通过细粒度策略限制到账户或 provider 级路径。
- 所有密钥读取行为必须审计。

## 11. 发布流水线设计

### 11.1 标准流水线

1. 代码提交触发 CI。
2. 执行单元测试、契约测试、静态分析、依赖漏洞扫描。
3. 构建镜像、生成 SBOM、签名并推送到 OCI Registry。
4. 渲染并校验 Helm values / 配置产物。
5. 更新环境 GitOps 仓库中的镜像版本与配置版本。
6. Argo CD 同步目标环境。
7. Argo Rollouts 执行金丝雀或蓝绿发布。
8. 发布后自动执行烟雾测试、探针检查、错误率和延迟检查。
9. 验证通过后完成 rollout promotion；失败则自动中止并回滚。

### 11.2 发布策略

| 场景 | 策略 |
| --- | --- |
| 普通 HTTP / gRPC 服务 | 金丝雀发布 |
| `api-gateway` | 金丝雀发布 + WebSocket 长连接排水 |
| Kafka consumer / worker | 分批滚动更新，控制并发重平衡 |
| 适配服务 | 保守滚动发布，确保旧实例优雅排空运行时会话 |
| 大版本数据结构升级 | 先扩展后切换的 Expand-Contract 模式 |

## 12. 数据库迁移与兼容发布

### 12.1 迁移原则

- 所有数据库迁移通过独立 `migration-job` 执行。
- 迁移必须满足向前兼容，至少覆盖一个发布窗口。
- 禁止把破坏性 DDL 与新代码同时一次性上线。

### 12.2 推荐流程

1. 先上线兼容旧代码的新字段或新表。
2. 上线新版本应用，同时兼容新旧字段。
3. 数据回填完成并验证后，移除旧逻辑。
4. 最后在独立窗口执行收缩型 DDL。

## 13. 伸缩与容量设计

### 13.1 自动伸缩

- `http-api` 根据 CPU、内存、请求并发、P95 延迟伸缩。
- `event-consumer` 根据 Kafka lag、处理耗时和失败率伸缩。
- `temporal-worker` 根据 task queue backlog、活动执行时间伸缩。
- `websocket` 根据连接数、发送队列深度和内存占用伸缩。

### 13.2 稳定性约束

- 所有核心 Deployment 必须配置 HPA、PDB、Topology Spread Constraints、反亲和策略。
- 核心服务至少保持 3 个副本，跨两个以上可用区分散。
- 适配服务若需会话粘性，粘性应通过 Redis 映射或外部状态引用实现，不依赖单实例本地内存。

## 14. 高可用与灾备设计

### 14.1 目标态

- `prod-primary` 为主生产区域，跨 3 AZ 高可用。
- `prod-dr` 为异地灾备区域，保持热备或温备。
- 业务写流量默认单主区域，避免多主写入造成事实冲突；边缘流量可全局路由到主区域。

### 14.2 数据复制

- MySQL 通过跨区域异步复制实现灾备同步。
- Kafka 通过 MirrorMaker 2 或等价机制复制关键 topic。
- 对象存储开启跨区域复制。
- OpenSearch、ClickHouse 采用异地快照或复制策略。

### 14.3 灾备目标

| 组件 | RPO | RTO |
| --- | --- | --- |
| MySQL 业务库 | <= 5 分钟 | <= 30 分钟 |
| Kafka 关键事件 | <= 5 分钟 | <= 30 分钟 |
| 对象存储 | <= 15 分钟 | <= 60 分钟 |
| 配置与密钥系统 | <= 5 分钟 | <= 30 分钟 |

### 14.4 切换规则

- 灾备切换必须是显式运营操作，不允许自动多主漂移。
- 切换前必须冻结发布、停止高风险写入并记录切换窗口。
- 切换后必须重新校验配置版本、密钥挂载、Kafka 消费位点和 Temporal namespace 可用性。

## 15. 安全设计

### 15.1 基础安全控制

- 所有镜像必须经过漏洞扫描、签名和准入校验。
- 集群内所有服务间调用默认启用 mTLS。
- Pod 默认使用非 root 用户、只读根文件系统、最小 Linux Capability。
- 敏感 namespace 需要单独网络策略、审计策略和节点隔离。

### 15.2 合规与审计

- 配置变更、密钥轮换、发布、回滚、灾备切换都必须留痕。
- 审计日志必须进入专用索引并设置不可篡改保留策略。
- 与券商、交易所、用户身份相关的敏感流量必须通过受控出网和日志脱敏策略处理。

## 16. 可观测性与发布守门

### 16.1 发布前守门

- 单元测试、契约测试、集成测试、Schema 兼容检查必须通过。
- 关键服务的 Helm 渲染、策略校验、网络策略校验必须通过。
- 镜像签名、SBOM 和漏洞门禁必须通过。

### 16.2 发布后守门

- 关键指标：错误率、P95/P99 延迟、Kafka lag、worker backlog、数据库连接占用、队列堆积、WebSocket 推送成功率。
- 任一守门指标超阈值，Rollout 必须自动暂停或回滚。

## 17. 回滚策略

### 17.1 应用回滚

- 通过 Argo Rollouts 回退到上一稳定版本。
- 回滚时优先保留配置版本不变，除非问题来自配置本身。

### 17.2 配置回滚

- 配置回滚通过 GitOps 仓库回退到前一提交并重新同步。
- 密钥回滚必须基于版本号切换，不得手工覆盖原值。

### 17.3 数据回滚

- 原则上不做数据库逻辑回滚，而采用向前修复。
- 若必须恢复数据，必须通过备份恢复、补偿脚本或业务修复作业完成，并留完整审计。

## 18. 运维禁令

- 手工登录生产 Pod 修改配置文件。
- 手工在生产环境执行 `kubectl edit` 作为常态发布手段。
- 应用主进程自带数据库迁移或初始化逻辑。
- 把适配服务与核心交易服务部署在同一隔离级别和同一出网策略下。

## 19. 结论

TradingClaw 的目标部署架构必须是“以 Kubernetes 为唯一正式运行平面、以 GitOps 为唯一正式变更入口、以 Vault 为唯一密钥真源、以服务网格和多可用区为高可用底座、以灾备区域承接跨区域恢复”的长期方案。后续所有服务上线、扩容、迁移、灰度、回滚和灾备演练都应围绕这套架构执行。
