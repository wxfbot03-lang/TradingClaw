# TradingClaw 运行启动与环境编排详细设计

## 1. 文档定位

- 本文档覆盖所有服务、后台作业、事件消费者、工作流 worker、适配器进程和一次性任务的统一启动方式与环境编排规则。
- 本文档解决“一个服务以什么角色启动”“启动顺序如何定义”“本地、测试、预发、生产如何保持一致”“健康检查和优雅停机如何统一”的问题。
- 本文档不描述最终生产集群拓扑和发布流水线细节，相关内容由 `部署与发布详细设计.md` 负责。

## 2. 设计目标

- 为所有服务定义统一的进程角色模型和启动入口。
- 保证启动行为可预测、可验证、可编排、可回滚。
- 让本地开发、集成测试、预发和生产尽可能共享同一份角色定义和启动协议。
- 避免“一个容器同时承担过多角色”“应用启动时偷偷建表、建 topic、跑迁移”等反模式。

## 3. 启动总原则

### 3.1 核心原则

- 一个可部署单元只承担一个主角色；必要时允许同一服务有多个 Deployment，但每个进程职责单一。
- 应用启动只负责装载配置、连接依赖、注册路由和声明健康状态；不负责隐式执行数据库迁移、Kafka topic 初始化或历史修复。
- 所有角色都必须通过统一引导器启动，并输出一致的结构化启动日志。
- 启动成功的标准不是进程已起来，而是 Readiness 条件满足并且必要依赖预检通过。

### 3.2 标准入口模型

所有 Python 服务统一通过一个引导入口启动：

```text
python -m tradingclaw.bootstrap --service <service_name> --role <role_name>
```

约束：

- 角色名必须来自固定枚举，不允许自由字符串。
- 引导器负责：配置装载、密钥解析、日志初始化、OpenTelemetry 初始化、依赖预检、应用装配、信号处理、健康探针注册。
- 业务模块只暴露装配函数，不直接在 `__main__` 中拼接启动逻辑。

## 4. 进程角色模型

### 4.1 标准角色枚举

| 角色 | 说明 |
| --- | --- |
| `http-api` | 对外 HTTP/JSON 服务 |
| `grpc-api` | 服务间 gRPC 服务 |
| `websocket` | WebSocket 连接与推送服务 |
| `event-consumer` | Kafka 事件消费者 |
| `temporal-worker` | Temporal workflow / activity worker |
| `scheduler` | 定时调度器或周期触发器 |
| `adapter-worker` | 通道适配运行时、回调处理、轮询回补 |
| `backfill-worker` | 历史回补与批处理作业 |
| `migration-job` | 数据库迁移作业 |
| `admin-job` | 一次性管理任务，如索引重建、topic 校验 |

### 4.2 角色责任边界

- `http-api` 只负责同步 API，不直接消费高吞吐事件。
- `grpc-api` 只负责同步服务间调用，不负责长期后台轮询。
- `event-consumer` 只消费事件并驱动领域逻辑，不对外提供监听端口。
- `temporal-worker` 只注册 workflow 和 activity，不暴露 HTTP API。
- `migration-job` 只在发布流程中执行，不作为常驻进程。

## 6. 标准启动生命周期

### 6.1 启动阶段

1. `BOOTSTRAP_INIT`：读取引导变量并锁定 `service`、`role`、`env`、`region`。
2. `CONFIG_LOAD`：加载渲染配置文件。
3. `SECRET_RESOLVE`：解析密钥引用并构建只读配置快照。
4. `OBSERVABILITY_INIT`：初始化结构化日志、指标、链路追踪。
5. `SCHEMA_VALIDATE`：执行 Pydantic + JSON Schema + 跨字段校验。
6. `DEPENDENCY_PREFLIGHT`：检查数据库、Redis、Kafka、Schema Registry、Temporal、对象存储、外部依赖可达性。
7. `APP_WIRE`：装配路由、消费者、workflow、连接池和资源池。
8. `READINESS_WAIT`：等待必要资源就绪，如 Kafka consumer group 分配、gRPC server bind 成功、workflow 注册完成。
9. `SERVING`：对外声明 readiness，并进入稳定运行状态。

### 6.2 阶段约束

- 任一阶段失败必须立即启动失败，避免服务以半残状态对外提供能力。
- `DEPENDENCY_PREFLIGHT` 只检查必要依赖；对可选依赖应降级标记，而不是阻断所有服务。
- `APP_WIRE` 阶段严禁执行业务副作用，例如扫描历史订单重建读模型。

## 8. 迁移与一次性任务规则

### 8.1 `migration-job` 规则

- 每个服务拥有独立迁移目录和独立迁移作业。
- 数据库迁移只能由发布流程触发，不允许应用主进程在启动时自动执行 Alembic upgrade。
- 迁移作业必须具备全局互斥锁，避免多实例并发执行。

### 8.2 `admin-job` 规则

- 建 topic、注册 Schema、重建索引、历史回补初始化等操作必须显式作为 `admin-job` 或平台初始化作业执行。
- `admin-job` 必须是幂等的，并记录执行 ID、目标环境、执行摘要和结果。

## 9. 本地、测试与生产启动模式

### 9.1 标准启动层级

| 模式 | 目标 | 典型内容 |
| --- | --- | --- |
| 单服务模式 | 开发单个服务 | 本地 Python 进程 + 共享依赖 |
| 域链路模式 | 联调某个业务域 | 若干服务 + 必要基础设施 |
| 全栈一致性模式 | 端到端集成验证 | 全部核心服务 + 平台依赖 |
| 集群一致性模式 | 预发与生产近似验证 | 与生产相同的 Helm / Kustomize 部署方式 |

### 9.2 统一控制命令设计

建议提供统一控制工具 `tcctl`，负责本地和集群启动编排：

```text
tcctl config render --env local --region local --service trade-gateway-service --role http-api
tcctl env up --profile core-trading
tcctl service start --service trade-gateway-service --role http-api
tcctl stack up --profile full-parity
tcctl health check --profile full-parity
```

设计要求：

- `tcctl` 只负责启动编排和诊断，不承载业务逻辑。
- 本地启动与 CI 集成测试都复用同一份服务描述文件。
- 集群环境不通过 `tcctl` 直接改线上资源，而是生成并验证部署产物。

### 9.3 本地依赖编排

- 本地开发可以使用容器化依赖，但依赖清单必须与生产平台冻结方案一致：MySQL、Redis、Kafka、Schema Registry、Temporal、对象存储模拟、OpenSearch、ClickHouse。
- 本地允许降低副本数和容量，不允许替换成完全不同的中间件品类。
- 本地开发若使用轻量替代实现，必须仅限单元测试，不得作为联调和集成环境基线。

## 10. 健康检查与可用性判定

### 10.1 探针定义

| 探针 | 语义 |
| --- | --- |
| `startupProbe` | 服务是否完成冷启动流程 |
| `readinessProbe` | 服务是否可接收流量或消费任务 |
| `livenessProbe` | 进程是否仍处于健康运行状态 |
| `dependencyProbe` | 关键依赖是否处于可接受范围 |

### 10.2 角色级 Readiness 条件

- `http-api`：监听端口成功、路由注册完成、数据库连接池可用、必要缓存和依赖可访问。
- `grpc-api`：端口监听成功、服务注册完成、反射和健康检查服务就绪。
- `event-consumer`：Kafka 分配完成、Schema 兼容检查通过、去重与幂等存储可用。
- `temporal-worker`：成功注册 workflow / activity，已订阅正确 task queue。
- `adapter-worker`：必要 egress、密钥、运行时上下文池和回补调度器均就绪。

### 10.3 降级与拒流

- 当关键依赖不可用时，服务应将 readiness 置为 false，而不是继续吞请求后再整体失败。
- `DEGRADED` 状态的业务语义由各领域维护，但基础设施层面的不可用必须先通过探针和入口拒流体现。

## 11. 优雅停机与重启

### 11.1 停机步骤

1. 接收 `SIGTERM`。
2. 立即将 readiness 置为 false，停止接收新流量或新分区分配。
3. HTTP / gRPC 等待在途请求完成。
4. Kafka consumer 提交安全 offset，并停止拉取新消息。
5. Temporal worker 停止领取新任务并等待当前任务完成或超时。
6. 关闭连接池、刷新指标、输出停机摘要日志。

### 11.2 约束

- 所有角色必须实现幂等停机。
- 不允许依赖 `kill -9` 或异常退出完成资源清理。
- 对于长任务，必须有上限超时和恢复机制，不能无限等待。

## 12. 启动失败与恢复策略

### 12.1 常见失败分类

| 类型 | 策略 |
| --- | --- |
| 配置缺失 | 立即失败，阻断启动 |
| 密钥解析失败 | 立即失败，触发告警 |
| 数据库不可达 | 阻断 `http-api`、`grpc-api`、`consumer` |
| Kafka 不可达 | 阻断 `consumer`，允许纯查询 API 视场景降级 |
| Schema 不兼容 | 立即失败，阻断发布 |
| 外部通道不可达 | 仅阻断依赖该通道的适配角色 |

### 12.2 自动恢复策略

- 对临时网络抖动采用指数退避重试，但重试窗口必须受限。
- 对配置或 Schema 级错误不得自动无限重试，必须等待配置修复或回滚。
- 对消费型角色可通过重启重新参与分区分配，但必须保证幂等消费。

## 14. 禁止反模式

- 一个进程同时提供 HTTP、gRPC、Kafka 消费和定时作业。
- 应用启动时自动执行数据库迁移、建 topic、修历史数据。
- 本地环境使用与生产完全不同的中间件品类，导致联调假象。
- 无统一引导器，每个服务自行处理日志、配置、信号和探针。

## 15. 结论

TradingClaw 的运行启动模型必须收敛为“统一引导入口 + 固定角色枚举 + 明确启动阶段 + 清晰依赖顺序 + 统一健康检查 + 优雅停机 + 环境一致性编排”的工程基线。只有这样，业务域的复杂状态机、事件流和恢复闭环才能建立在可预测的运行时之上。
