## 2. 模块职责

### 2.1 `audit-risk-service`

- 风险规则匹配与裁决。
- 审计日志归集。
- 风险事件记录、人工复核、闭环处置。

### 2.2 `notification-service`

- 系统通知、策略通知、风险通知、异常告警发送。
- 通知投递结果回写和失败重试。

## 3. 领域边界

- 风控模块提供裁决，不直接修改交易或策略主状态。
- 审计模块记录关键动作，但不成为业务事实源。
- 通知模块消费事件并投递，不反向持有业务主状态。

## 5. 核心流程

### 5.1 交易前风控裁决

1. 接收交易网关或策略运行时的风险校验请求。
2. 校验身份、账户归属、额度、状态、参数越界和风险规则。
3. 返回通过、拒绝或需要人工复核。
4. 对命中规则的请求记录风险事件。
5. 若需要人工复核，发布 `risk.review_required` 事件，并由复核结论驱动原请求继续或终止。

首批规则集要求：

- 账户状态与会话有效性规则
- 单笔下单金额/数量上限规则
- 日累计交易额度规则
- 杠杆与保证金约束规则
- 高频提交限流规则
- 标的黑白名单和交易时段规则

规则冲突处理：

- 按 `priority` 从高到低匹配。
- 同一请求命中多条规则时，`REJECT` 优先级高于 `REVIEW`，`REVIEW` 高于 `ALLOW`。
- 若多条同级规则同时命中，以最严格结论为准，并在事件中记录主命中规则和附加命中列表。

### 5.2 审计旁路消费

1. 消费登录、下单、撤单、策略启动、AI 工具调用等关键事件。
2. 写入审计日志。
3. 关联 `trace_id`、`request_id`、业务主键。

### 5.3 人工复核闭环

1. 高风险事件进入 `PENDING_REVIEW`。
2. 人工或自动复核形成结论。
3. 更新到 `CONFIRMED`、`DISMISSED` 或 `MITIGATED`。
4. 最终进入 `CLOSED`。
5. 发布 `risk.review_completed` 事件，通知交易网关或策略运行时继续原请求或终止流程。
6. 复核通过后的恢复统一通过工作流 signal 或显式恢复命令触发，不允许消费者自行拼装原请求重放。

### 5.4 通知投递

1. 接收 `notification.send_requested` 集成命令事件。
2. 选择通知渠道和模板。
3. 创建 `Notification` 记录并发布 `notification.created` 领域事实事件。
4. 执行投递并记录结果。
5. 发布 `notification.delivered` 或失败重试记录。

## 6. 状态机

### 6.1 风险事件状态机

- `DETECTED`
- `PENDING_REVIEW`
- `CONFIRMED`
- `DISMISSED`
- `MITIGATED`
- `CLOSED`

规则：

- 命中高风险规则必须进入 `DETECTED`。
- 需要人工介入时进入 `PENDING_REVIEW`。
- 只有处置完成后才能进入 `MITIGATED` 再闭环。

## 7. 数据设计

核心表：

- `audit_logs`
- `risk_rules`
- `risk_events`
- `manual_reviews`
- `notifications`
- `notification_deliveries`

设计要点：

- `audit_logs`、`risk_rules`、`risk_events`、`manual_reviews`、`notifications`、`notification_deliveries` 必须落 MySQL，保证审计、风控和通知闭环具备完整追溯能力。
- Redis 用于规则命中热点缓存、通知发送去重键、短期告警聚合和人工复核任务的暂存索引，不作为合规留痕依据。
- 任何告警、裁决、复核、发送结果都必须以 MySQL 记录为准，Redis 失效不能影响历史审计查询。
- 建议索引：`audit_logs(actor_id, created_at)`、`audit_logs(resource_type, resource_id, created_at)`、`risk_events(status, severity, created_at)`、`notification_deliveries(notification_id, channel)`。
- `audit_logs` 与 `risk_events` 应按时间维度预留归档策略，避免高增长明细长期挤压在线查询表。
- 默认值建议：`risk_events.status = DETECTED`，`notifications.status = CREATED`，`notification_deliveries.status = PENDING`（对应全局“通知投递尝试状态枚举”）。
- 空值规则：`decision`、`comment`、`payload_ref` 可空；`risk_event_id`、`resource_type`、`resource_ref`、`action_type`、`status` 默认不可空，`resource_id` 在预创建资源场景可延后补齐。
- 删除策略：风控事件、审计日志、通知记录默认不物理删除；通知失败重试通过投递记录追加实现，不覆盖历史结果。
- 审计要求：人工复核、风险裁决、通知模板变更建议记录 `created_by`、`updated_by`、`trace_id`。
- `risk_events` 必须持久化 `rule_code`、`rule_version`、`resource_ref` 与恢复引用，保证命中来源和复核恢复链路可追溯。
- 多规则命中时，`rule_code` / `rule_version` 记录主命中规则，附加命中规则统一落在 `matched_rules` 中。
- `notification.send_requested` 必须携带稳定幂等键；通知域基于 `idempotency_key` 或 `business_key + notification_type + channel` 执行去重。

## 8. 事件设计

核心事件：

- `risk.rule_triggered`
- `risk.review_required`
- `risk.review_completed`
- `notification.send_requested`
- `notification.created`
- `notification.delivered`

恢复载荷约束：

- `risk.review_completed` 至少应携带 `risk_event_id`、`resource_type`、`resource_id`、`resource_ref`、`decision`。
- 如资源处于可恢复流程中，事件还应携带 `workflow_id`、原始 `idempotency_key` 与 `resume_token` 等恢复引用。
- `notification.created`、`notification.delivered` 仅允许 `notification-service` 发布；`notification.send_requested` 仅作为上游到通知域的集成命令事件。

## 9. 接口设计

### 9.1 gRPC 服务

- `RiskDecisionService`
- `AuditService`

#### 9.1.1 `RiskDecisionService.Evaluate`

请求字段：`resource_type`、`resource_ref`、`resource_id`、`user_id`、`account_id`、`action_type`、`payload`

响应字段：`allowed`、`decision`、`reason_code`、`review_required`、`risk_event_id`

语义约束：

- `resource_ref` 必填，作为风控命中、审计检索和恢复链路的稳定引用。
- `resource_id` 在资源已分配业务主键时必填；交易下单场景必须先预分配 `order_id`，再以该 `order_id` 调用风控。
- `allowed = true` 且 `review_required = false`：调用方可继续执行。
- `allowed = false` 且 `review_required = false`：调用方应立即拒绝。
- `review_required = true`：调用方必须保留原请求并进入待复核状态，不得继续执行通道写操作。

#### 9.1.2 `AuditService.Record`

请求字段：`actor_id`、`action_type`、`resource_type`、`resource_ref`、`resource_id`、`trace_id`、`payload_ref`

响应字段：`audit_log_id`、`recorded`

## 10. 实施要点

- 风控裁决必须在交易和策略执行前接入，不允许前端或 CLI 替代。
- 审计日志禁止静默删除。
- 通知模块必须支持失败重试、幂等和结果回写。
