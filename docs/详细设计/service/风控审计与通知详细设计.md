# TradingClaw 风控审计与通知详细设计

## 1. 文档定位

- 本文档覆盖 `audit-risk-service` 与 `notification-service`，定义风险裁决、人工复核、审计留痕和通知投递的统一治理设计。
- 本模块为交易、策略、AI 和平台运营提供前置治理与旁路治理能力，但不越权持有交易或策略主状态。
- 风险事件、审计记录和通知记录都以可追溯、可恢复、可审计为核心要求。

## 2. 模块定位

### 2.1 `audit-risk-service`

- 提供统一风险规则匹配、裁决和人工复核闭环。
- 记录风控事件、治理动作和审计日志。
- 向交易与策略主线返回允许、拒绝或人工复核结果。

### 2.2 `notification-service`

- 提供系统通知、策略通知、风险通知和异常告警投递能力。
- 管理通知主记录、投递结果、失败重试和去重逻辑。
- 通过标准事件对外暴露通知事实，不反向持有业务主状态。

## 3. 领域边界

- 风控模块提供裁决与复核结论，不直接修改交易订单状态或策略实例状态。
- 审计模块只记录关键动作和链路引用，不成为业务事实源。
- 通知模块消费事件并执行投递，不参与业务决策。
- 恢复交易或策略原流程的动作由原业务工作流执行，治理域只提供恢复所需的标准引用。

## 5. 治理主线

### 5.1 风控裁决主线

风控裁决是交易、策略和高风险治理动作的统一前置检查入口。

流程如下：

1. 接收交易网关、策略运行时或其他治理场景发来的风险校验请求。
2. 校验身份、账户归属、额度、状态、参数越界、交易会话和风险规则。
3. 按规则优先级与决策优先级计算最终裁决。
4. 返回 `ALLOW`、`REJECT` 或 `REVIEW` 结果，并在命中时记录 `RiskEvent`。
5. 若结论为人工复核，则发布 `risk.review_required` 事件，由原业务流程转入等待状态。

首批规则集要求：

- 账户状态与会话有效性规则
- 单笔下单金额或数量上限规则
- 日累计交易额度规则
- 杠杆与保证金约束规则
- 高频提交限流规则
- 标的黑白名单和交易时段规则

规则冲突处理：

- 按 `priority` 从高到低匹配。
- 同一请求命中多条规则时，`REJECT` 优先级高于 `REVIEW`，`REVIEW` 高于 `ALLOW`。
- 同级规则同时命中时，以最严格结论为准，并在事件中记录主命中规则和附加命中列表。

### 5.2 规则治理主线

风险规则不仅是裁决输入，也是一套受控发布和版本治理的治理资产。

要求如下：

- 首批规则必须形成可枚举清单和默认启停策略，不能只保留抽象框架。
- 规则以 `rule_code + rule_version` 作为稳定版本键。
- 风控缓存刷新必须支持版本化加载；规则变更后 30 秒内完成全节点刷新。
- 额度口径优先取交易网关标准化订单事实与账户余额快照，不直接依赖适配侧原始读数。

### 5.3 人工复核与恢复主线

人工复核用于处理高风险但可由人工判定是否继续的请求。

流程如下：

1. 高风险事件进入 `PENDING_REVIEW`。
2. 人工或自动复核形成结论。
3. 风险事件更新为 `CONFIRMED`、`DISMISSED` 或 `MITIGATED`。
4. 处理完成后进入 `CLOSED`。
5. 发布 `risk.review_completed`，通知交易网关或策略运行时恢复或终止原流程。

恢复规则：

- 复核通过后的恢复统一通过工作流 signal 或显式恢复命令触发。
- 治理域只提供恢复引用，不允许消费者自行拼装原始请求重放。
- `risk.review_completed` 至少必须携带 `risk_event_id`、`resource_type`、`resource_id`、`resource_ref`、`decision`。
- 若资源处于可恢复流程中，还必须携带 `workflow_id`、原始 `idempotency_key`、`resume_token` 等恢复引用。

### 5.4 审计旁路主线

审计模块负责将关键动作沉淀为可检索、可回放的治理日志。

流程如下：

1. 消费登录、下单、撤单、策略启动、AI 工具调用和复核操作等关键事件。
2. 关联 `trace_id`、`request_id`、业务主键和资源引用。
3. 写入 `AuditLog`，作为旁路留痕事实。

约束：

- 审计日志只追加不静默删除。
- 审计记录中的 `resource_ref` 应支持跨系统检索同一业务链路。

### 5.5 通知命令与通知事实主线

通知域将“请求发送通知”与“通知事实产生”“通知投递结果”明确拆分，避免语义混用。

流程如下：

1. 接收 `notification.send_requested` 集成命令事件。
2. 根据业务类型、目标用户和模板策略选择通知渠道与模板。
3. 创建 `Notification` 主记录，并发布 `notification.created` 领域事实事件。
4. 执行投递，记录 `DeliveryAttempt`。
5. 成功时发布 `notification.delivered`；失败时根据重试策略追加新的投递记录。

统一语义约束：

- `notification.send_requested` 只表示“上游要求通知域开始处理”，不表示通知已创建。
- `notification.created` 表示通知主资源已登记。
- `notification.delivered` 表示某次投递已完成并形成结果事实。

## 6. 状态机

### 6.1 风险事件状态机

- `DETECTED`
- `PENDING_REVIEW`
- `CONFIRMED`
- `DISMISSED`
- `MITIGATED`
- `CLOSED`

规则：

- 命中高风险规则后必须进入 `DETECTED`。
- 需要人工介入时进入 `PENDING_REVIEW`。
- 完成处置后才能进入 `MITIGATED` 或其他终态，再进入 `CLOSED`。

## 7. 数据设计

核心表：

- `audit_logs`
- `risk_rules`
- `risk_events`
- `manual_reviews`
- `notifications`
- `notification_deliveries`

设计要点：

- `audit_logs`、`risk_rules`、`risk_events`、`manual_reviews`、`notifications`、`notification_deliveries` 必须落 MySQL，保证治理闭环具备完整追溯能力。
- Redis 用于规则命中热点缓存、通知去重键、短期告警聚合和人工复核任务索引，不作为合规留痕依据。
- 告警、裁决、复核和发送结果都必须以 MySQL 记录为准，Redis 失效不能影响历史查询。
- `audit_logs` 与 `risk_events` 应按时间维度预留归档策略，避免高增长明细长期挤压在线表。
- 风险事件、审计日志和通知记录默认不物理删除；通知失败重试通过投递记录追加，不覆盖历史结果。
- 人工复核、风险裁决和通知模板变更建议记录 `created_by`、`updated_by`、`trace_id`。

## 8. 事件设计

核心事件：

- `risk.rule_triggered`
- `risk.review_required`
- `risk.review_completed`
- `notification.send_requested`
- `notification.created`
- `notification.delivered`

事件约束：

- `notification.created`、`notification.delivered` 只能由 `notification-service` 发布。
- `notification.send_requested` 只作为上游到通知域的集成命令事件。
- `risk.review_completed` 必须提供足够的恢复引用，使原业务工作流可以无歧义地恢复或终止。

## 9. 接口设计

### 9.1 gRPC 服务

- `RiskDecisionService`
- `AuditService`

#### 9.1.1 `RiskDecisionService.Evaluate`

请求字段：`resource_type`、`resource_ref`、`resource_id`、`user_id`、`account_id`、`action_type`、`payload`

响应字段：`allowed`、`decision`、`reason_code`、`review_required`、`risk_event_id`

语义约束：

- `resource_ref` 必填，作为风控命中、审计检索和恢复链路的稳定引用。
- `resource_id` 在资源已分配业务主键时必填；交易下单场景必须先预分配 `order_id`，再以该 `order_id` 调用风控。
- `allowed = true` 且 `review_required = false` 时，调用方可继续执行。
- `allowed = false` 且 `review_required = false` 时，调用方应立即拒绝。
- `review_required = true` 时，调用方必须保留原请求并进入待复核状态，不得继续执行通道写操作。

#### 9.1.2 `AuditService.Record`

请求字段：`actor_id`、`action_type`、`resource_type`、`resource_ref`、`resource_id`、`trace_id`、`payload_ref`

响应字段：`audit_log_id`、`recorded`

## 10. 实施要点

- 风控裁决必须在交易和策略执行前接入，不允许由前端或 CLI 替代。
- 审计日志禁止静默删除。
- 通知模块必须支持失败重试、幂等和结果回写。
- 人工复核恢复链路必须与工作流恢复机制对齐，避免出现治理域和业务域各自维护一套恢复语义。
