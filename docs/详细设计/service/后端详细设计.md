## 3. 总体架构

### 3.1 架构风格

系统采用以下组合式架构：

- DDD：每个业务域拥有自己的主数据、状态机和事件语义。
- 事件驱动微服务：跨域传播以标准事件和适配事件完成，避免共享数据库。
- 工作流编排：长流程、补偿、人工复核恢复交给工作流层管理。
- 插件化适配器：券商、交易所、行情源、资讯源以及外部模型服务通过适配器接入。

### 3.2 分层视图

系统分为五层：

1. 接入层：`api-gateway`、鉴权、限流、能力探测、WebSocket 接入。
2. 领域服务层：身份账户、市场数据、统一交易、策略、治理、AI 等业务服务。
3. 事件与编排层：事件总线、Schema 管理、工作流引擎、回放与补偿机制。
4. 数据与平台层：MySQL、Redis、时序存储、搜索、对象存储、密钥管理。
5. 外部适配层：券商、交易所、行情源、资讯源和外部模型服务（如 LiteLLM）。

### 3.3 关键主线

- 身份账户主线：用户登录 -> 会话有效 -> 账户归属 -> 账户能力投影 -> 默认交易会话引用。
- 交易主线：统一交易网关接收命令 -> 风控裁决 -> 路由适配域 -> 回报归一 -> 持仓余额刷新。
- 策略主线：策略配置 -> 运行时触发 -> 纯计算生成计划 -> 风控 -> 交易执行 -> 快照恢复。
- 治理主线：审计旁路记录 -> 风险命中 -> 人工复核 -> 恢复或终止 -> 通知投递。

## 4. 核心边界与全局约束

### 4.1 领域边界

- `identity-service` 持有用户身份和登录会话事实。
- `account-service` 持有账户主数据、绑定关系、账户能力投影、默认交易会话引用。
- `market-service` 独占 `instrument_id` 主权。
- `trade-gateway-service` 独占统一交易会话、订单、成交、持仓、余额和交易规则读模型。
- `securities-trading-service`、`crypto-trading-service` 仅持有适配侧原始事实。
- `strategy-service`、`strategy-runtime-service`、`portfolio-calculation-service` 共同构成策略域闭环。
- `audit-risk-service`、`notification-service` 提供治理能力，不越权修改交易和策略主状态。

### 4.2 主数据与引用规则

- 主数据 owner 只能有一个，引用不代表主权转移。
- `default_trading_session_id` 由账户域持有，但只允许引用交易域中状态为 `AVAILABLE` 的会话。
- `instrument_id` 是跨域稳定主键，`symbol + market` 仅保留兼容定位能力。
- 与订单恢复相关的事实统一使用 `order_id` 作为主键、主分区键和回放主锚点。

### 4.3 事件与工作流规则

- 领域事实通过标准事件传播，适配域只发布原始事件或返回 gRPC 结果。
- 所有事件必须使用统一信封，并在 `事件字段字典.md` 中登记。
- 事件总线在实现前冻结，推荐采用 Kafka + Schema Registry。
- 默认交付语义按 at-least-once 设计，靠幂等键、去重表和回放规范保证最终一致。
- 工作流负责长流程、补偿、人工复核恢复，不充当最终事实源。

### 4.4 数据基线

- MySQL 是业务最终事实源，所有可审计、可恢复、可回放的核心对象必须能从 MySQL 恢复。
- Redis 只保存可失效、可重建的派生状态，如热点缓存、幂等键、限流计数、分布式锁和短期索引。
- 高吞吐行情与指标进入时序/分析存储。
- 资讯检索和日志检索进入搜索引擎。
- OCR 图片、原始报文、长文本上下文和审计附件进入对象存储。

### 4.5 接口基线

- 对客户端统一暴露 HTTP/JSON 与 WebSocket。
- 服务间高频同步调用优先使用 gRPC。
- 跨域异步协作使用事件。
- 长流程活动和补偿步骤通过工作流活动暴露。
- 命令接口统一要求 `X-Idempotency-Key`。

### 4.6 状态与枚举基线

- 状态机由领域层集中维护，不允许控制器、消费者或前端自行派生新状态。
- 外部通道状态必须先映射为统一状态，再映射为展示状态。
- `状态字段枚举表.md` 是 `status`、`trading_session_status`、`adapter_session_status`、`time_in_force`、`severity` 等字段的唯一枚举源。

## 5. 技术与平台基线

### 5.1 服务内技术栈

- 后端主开发语言统一为 Python。
- HTTP / WebSocket 接入使用 FastAPI。
- 数据访问使用 SQLAlchemy 2.x + Alembic。
- 接口模型使用 Pydantic v2。
- Redis 接入使用 `redis-py`。
- gRPC 使用 `grpcio` + proto。
- 工作流编排使用 Temporal Python SDK。
- 测试统一使用 `pytest`。

### 5.2 基础设施冻结要求

- 消息中间件、时序库、对象存储、密钥管理方案必须在接口冻结前确定。
- 市场时序数据、OCR 原图和大模型上下文等大对象不得直接落事务主库。
- 搜索、时序、对象存储更换属于平台演进，不应在业务实现阶段边开发边切换。

## 6. 模块拆分原则

- 一个文档描述一个可独立排期和交付的功能模块。
- 模块文档只保留本域职责、流程、数据、事件、接口和依赖。
- 跨模块通用规则统一在本文档和全局治理文档维护。
- 扩展新策略类型、新交易通道、新供应商时，应优先扩展模块和字典，不应修改全局边界。

### 7.2 第二批：市场数据底座

- `行情与资讯详细设计.md`

目标：形成统一标的主数据、行情、历史数据、指标和资讯供给能力。

### 7.3 第三批：交易与治理主链路

- `交易网关详细设计.md`
- `证券交易详细设计.md`
- `数字资产交易详细设计.md`
- `风控审计与通知详细设计.md`

目标：打通统一交易会话、下单、回报归一、风控裁决、人工复核和通知命令链路。

- `智能能力与治理详细设计.md`

目标：补齐 AI、OCR、能力治理及更完整的运营治理能力。
