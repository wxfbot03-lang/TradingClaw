# TradingClaw 事件字段字典

## 1. 文档定位

- 本文档统一 `docs/详细设计/service/` 下领域事件、集成命令事件和工作流相关事件的字段口径。
- 目标是让事件信封、事件 payload、分区键、回放与审计字段在各模块之间保持一致。
- 本文档补充 `API字段字典.md`，重点覆盖事件专用约束，而不是重复描述所有通用接口字段。

## 2. 事件设计原则

- 事件字段统一使用 `snake_case`。
- 事件必须区分“信封字段”和“业务 payload 字段”。
- 同一业务事实只允许一个主发布方。
- 同一聚合根相关事件必须具备稳定分区键，避免乱序消费。
- 事件总线平台必须在实现前冻结；推荐采用 Kafka + Schema Registry 作为主事实事件总线。
- 默认交付语义按 at-least-once 设计，并通过幂等消费、去重表和回放规范保证最终一致。

## 3. 事件信封

### 3.1 标准信封字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `event_id` | string | 是 | 事件唯一标识 |
| `event_type` | string | 是 | 事件类型，如 `order.created` |
| `event_version` | integer | 是 | 事件 schema 版本 |
| `occurred_at` | string(datetime) | 是 | 业务事实发生时间 |
| `published_at` | string(datetime) | 是 | 消息发布时间 |
| `producer` | string | 是 | 发布服务标识 |
| `trace_id` | string | 否 | 链路追踪 ID |
| `request_id` | string | 否 | 发起请求 ID |
| `workflow_id` | string | 否 | 关联工作流 ID |
| `aggregate_type` | string | 是 | 聚合根类型，如 `order`、`strategy` |
| `aggregate_id` | string | 是 | 聚合根 ID |
| `tenant_id` | string | 否 | 多租户预留字段 |
| `partition_key` | string | 是 | 分区键 |
| `payload` | object | 是 | 业务负载 |
| `meta` | object | 否 | 扩展元数据 |

### 3.2 元数据字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `meta.retry_count` | integer | 已重试次数 |
| `meta.source_env` | string | 来源环境，如 `prod`、`staging` |
| `meta.schema_name` | string | schema 名称 |
| `meta.schema_ref` | string | schema 引用或版本地址 |
| `meta.replay` | boolean | 是否来自回放 |
| `meta.dlq` | boolean | 是否来自死信队列 |
| `meta.reason_code` | string | 与失败相关的结构化原因码 |

## 4. 分区与路由规则

### 4.1 核心路由字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `partition_key` | string | 用于 topic 分区与顺序消费 |
| `aggregate_type` | string | 聚合根类型 |
| `aggregate_id` | string | 聚合根唯一 ID |
| `producer` | string | 发布方服务名称 |
| `event_type` | string | 事件类型，用于消费者路由 |

### 4.2 分区建议

- 用户事件按 `user_id`
- 账户事件按 `account_id`
- 行情事件按 `instrument_id`
- 订单事实事件按 `order_id`
- `execution.report_received` 按 `order_id`
- 与订单恢复相关的 `risk.review_completed` 按 `order_id`
- 策略事件按 `strategy_instance_id`
- 资讯事件按 `news_id`
- 通知事实事件按 `notification_id`

## 13. 使用规则

- 新增事件类型时，必须先定义信封字段、payload 字段、分区键，再进入模块文档。
- 同一字段如已在 `API字段字典.md` 定义，语义必须保持一致，不得另起一套命名。
- 事件 payload 应尽量围绕聚合根和消费目的最小化设计，不应塞入无关字段。
- 若事件字段语义变更，必须同步修改事件 schema、模块文档和回放审计逻辑。
