# TradingClaw 工程目录与文件树详细设计

## 1. 文档定位

- 本文档定义 TradingClaw 仓库的目标目录结构、服务骨架模板、运维目录、部署目录、脚本目录和生成产物边界。
- 本文档回答“代码应该放在哪里、配置和部署文件如何组织、不同服务如何保持同构”的问题。
- 本文档是平台工程基线，任何新服务、新作业、新部署单元都必须按本文档落位，禁止自由发挥目录结构。

## 2. 设计目标

- 让所有服务具备统一、可预测的工程骨架，降低跨服务认知成本。
- 把业务代码、平台代码、部署清单、配置清单、脚本工具、测试资源和生成产物明确隔离。
- 支持长期多服务扩张，而不出现“每个服务一个风格”的仓库退化。
- 为自动生成工具、脚手架、CI、GitOps 和审计规则提供稳定路径约定。

## 3. 总体原则

### 3.1 单仓原则

- TradingClaw 采用 Monorepo 作为唯一正式工程仓库。
- 所有后端服务、共享库、协议、配置、部署模板、脚本和设计文档统一收敛到同一仓库。
- 生成产物、运行日志、本地缓存和私有临时文件不得进入正式源目录。

### 3.2 分层原则

- `apps/` 只承载可部署服务和作业。
- `packages/` 只承载共享库、SDK、公共组件和生成代码。
- `ops/` 只承载配置、环境清单、脚本、平台描述和运行资产模板。
- `deploy/` 只承载与 Kubernetes / GitOps / Helm / Rollouts 直接相关的部署材料。
- `docs/` 只承载设计文档、操作文档和契约文档。

### 3.3 禁止事项

- 禁止把 Helm、Kustomize、临时脚本散落到各服务目录根部。
- 禁止在 `service/` 目录中直接存放环境特定 `.env`、私钥或运行日志。
- 禁止不同服务采用不同骨架，例如一个有 `domain/`，另一个只有 `src/`。

## 4. 目标仓库根目录

```text
TradingClaw/
  apps/
  packages/
  deploy/
  ops/
  docs/
  tools/
  tests/
  .github/
  Makefile
  pyproject.toml
  README.md
```

说明：

- `apps/` 存放所有服务与后台作业。
- `packages/` 存放共享库、proto、schema、SDK 和工程基建库。
- `deploy/` 存放 Helm chart、Argo CD Application、Rollout、NetworkPolicy 等正式部署清单。
- `ops/` 存放配置源、运行 profiles、脚本、演练模板和资产基线。
- `tools/` 存放仓库级 CLI、代码生成器、校验器。
- `tests/` 存放跨服务集成测试、契约测试和端到端测试。

### 5.3 服务目录职责

| 目录 | 说明 |
| --- | --- |
| `bootstrap/` | 角色装配、入口注册、生命周期钩子 |
| `domain/` | 聚合、实体、值对象、领域服务、状态机 |
| `app/` | 应用服务、命令、查询、编排协调 |
| `infra/` | 数据库、缓存、消息、对象存储、外部客户端 |
| `interfaces/` | HTTP、gRPC、事件消费者、worker 暴露层 |
| `module/` | 模块组合与依赖装配 |
| `observability/` | 日志、指标、trace、告警埋点封装 |
| `security/` | 鉴权、脱敏、加解密适配、权限检查 |
| `settings/` | 本服务配置模型与校验逻辑 |
| `migrations/` | Alembic 迁移脚本 |
| `schemas/` | JSON Schema、事件 Schema、配置 Schema 片段 |

### 5.4 角色文件约定

```text
src/.../bootstrap/
  roles/
    http_api.py
    grpc_api.py
    event_consumer.py
    temporal_worker.py
    scheduler.py
    adapter_worker.py
    migration_job.py
```

规则：

- 每个角色文件只负责一个角色装配。
- 所有角色最终由统一 bootstrap 调用，不允许直接从业务模块启动服务器。

## 6. `packages/` 目录设计

### 6.1 目标结构

```text
packages/
  platform-core/
  platform-bootstrap/
  platform-observability/
  platform-security/
  platform-config/
  platform-messaging/
  platform-testing/
  contracts-api/
  contracts-events/
  contracts-proto/
  trading-sdk/
  tooling-tcctl/
```

### 6.2 共享包职责

- `platform-core`：公共异常、Result 模型、通用类型、时间与 ID 工具。
- `platform-bootstrap`：统一启动引导器、角色注册器、生命周期状态机。
- `platform-observability`：日志、metrics、tracing 初始化与中间件。
- `platform-security`：JWT、mTLS 辅助、脱敏、密钥引用解析。
- `platform-config`：配置渲染、Schema 校验、Settings 基类。
- `platform-messaging`：Kafka Producer/Consumer、幂等、Schema Registry 访问器。
- `platform-testing`：测试 fixture、容器化依赖、契约测试工具。
- `contracts-api`：共享 API 字段模型与错误响应模型。
- `contracts-events`：标准事件信封与 payload 模型。
- `contracts-proto`：gRPC proto 定义和生成代码。
- `tooling-tcctl`：统一控制工具源码。

## 7. `deploy/` 目录设计

### 7.1 目标结构

```text
deploy/
  charts/
    tradingclaw-service/
    tradingclaw-worker/
    tradingclaw-job/
    platform-dependencies/
  argocd/
    applications/
    appsets/
    projects/
  rollouts/
  policies/
    network/
    pod-security/
    kyverno/
  gateways/
  observability/
  environments/
    dev-integration/
    qa/
    staging/
    prod-primary/
    prod-dr/
```

### 7.2 设计规则

- 正式部署清单必须以环境为入口组织，而不是以个人习惯组织。
- chart 只承载模板，不承载真实环境密钥或未渲染产物。
- `environments/` 只承载各环境引用的 values、应用清单组合和 policy 绑定。

### 7.3 Chart 分层

| Chart | 用途 |
| --- | --- |
| `tradingclaw-service` | HTTP / gRPC / WebSocket 类 Deployment |
| `tradingclaw-worker` | Consumer / Worker / Scheduler 类 Deployment |
| `tradingclaw-job` | Migration / Admin / Backfill 类 Job / CronJob |
| `platform-dependencies` | 非云托管平台组件或环境级依赖 |

## 8. `ops/` 目录设计

### 8.1 目标结构

```text
ops/
  config/
    schema/
    shared/
    environments/
    services/
    runtime/
  runtime/
    profiles/
    manifests/
    health/
  scripts/
    bootstrap/
    lint/
    render/
    release/
    drill/
  runbooks/
  drills/
  inventories/
  assets/
```

### 8.2 子目录职责

- `config/`：所有非敏感配置源、Schema、渲染产物索引。
- `runtime/profiles/`：本地和 CI 使用的启动 profile 定义。
- `runtime/manifests/`：`tcctl` 读取的服务角色编排描述。
- `runtime/health/`：健康检查清单和依赖探针定义。
- `scripts/`：仓库级工程脚本，按用途分类。
- `runbooks/`：发布、回滚、故障处置、切换操作规程。
- `drills/`：演练脚本、场景描述、验收记录模板。
- `inventories/`：环境、区域、外部端点、topic、队列、DNS 等资产清单。
- `assets/`：示意图源文件、证书模板、策略模板等非代码资产。

### 9.1 目标结构

```text
docs/
  详细设计/
    service/
    platform/
  runbooks/
  adr/
  api/

```

规则：

- `runbooks/` 聚焦可执行操作规程，不与设计文档混放。

## 11. `tools/` 目录设计

```text
tools/
  tcctl/
  codegen/
  schema-check/
  config-lint/
  release-audit/
```

规则：

- `tools/` 只放仓库级工具，不放业务脚本。
- 任何长期存在且会被 CI 或多人复用的脚本，都应从 `ops/scripts/` 或个人 shell 提升到 `tools/`。

## 12. 生成产物与禁止入库项

### 12.1 允许入库的生成产物

- 经评审确认必须版本化的 proto 生成代码。
- 前后端契约生成的只读 SDK，若发布链路依赖锁版本。

### 12.2 禁止入库项

- 本地 `.env`
- 渲染后的生产配置
- 运行日志
- 进程 PID 文件
- 本地数据库文件
- 临时导出的密钥与证书
- 调试抓取的原始报文与敏感截图

## 13. 新服务接入模板

新增服务时，必须至少同时新增：

1. `apps/<service>/` 服务骨架
2. `ops/config/services/<service>/` 配置源
3. `deploy/environments/*/` 环境引用
4. `packages/contracts-proto/` 或 `packages/contracts-events/` 契约更新
5. `docs/详细设计/service/` 对应详细设计文档

## 14. 禁止反模式

- 直接在服务目录下混放 `Dockerfile`、`helm/`、`tmp/`、`notes.md`、`local.env`
- 在多个地方重复保存同一份 proto、事件 Schema 或配置字段说明
- 把平台脚本放进个人目录后长期作为正式流程依赖

## 15. 结论

TradingClaw 的工程目录必须收敛为“`apps` 承载服务、`packages` 承载共享库、`deploy` 承载正式部署、`ops` 承载运行资产、`docs` 承载文档、`tools` 承载仓库级工具”的单仓结构。后续所有服务与平台能力都应在这套结构内扩展，而不是继续增加例外。
