# TradingClaw 后端需求文档

## 1. 文档说明

### 1.1 文档目标
- 明确 TradingClaw 后端的业务需求范围与交付标准。
- 为产品、开发、测试提供统一需求基线，降低实现偏差。
- 作为迭代验收依据，确保核心业务能力可验证。

### 1.2 适用范围
- 本文档覆盖后端业务能力需求与验收要求。
- 本文档不包含界面设计、技术架构设计、数据库设计、接口实现细节等设计阶段内容。

## 2. 产品目标

### 2.1 产品目标
- 为 openclaw 智能体提供好用的交易工具能力，覆盖行情、交易、策略、资讯、智能问答等核心场景。
- 保障核心链路在异常情况下可感知、可恢复、可追踪。
- 通过统一能力模型支撑多交易通道和策略类型扩展。

## 3. 业务范围与总体要求

### 3.1 业务范围
- 用户与会话管理
- 资产行情与指标服务
- 新闻资讯与影响力分析
- 证券交易能力
- 数字资产交易能力
- 策略管理与执行引擎
- 统一交易抽象层
- 智能问答与工具调用
- 应用入口与基础能力
- 交易计划能力
- 前端文档中心供给边界说明

### 3.2 总体业务要求
- 系统必须具备统一入口与统一服务编排能力。
- 系统必须支持配置驱动，能够动态读取模型、交易及运行参数。
- 所有关键业务失败必须返回明确状态与可读原因。

### 3.3 安全与合规要求
- 必须具备恶意请求识别与拦截能力。
- 必须保证身份认证、会话校验、操作主体一致性校验全链路生效。
- 日志和响应中不得暴露密钥、令牌、密码等敏感信息。
- 必须防止参数注入、越权访问等非法请求行为。

### 3.4 稳定性要求
- 高频查询场景必须支持缓存以降低外部依赖压力。
- 外部依赖异常时，应优先返回可用缓存并标注数据状态。
- 服务退化时应保证核心能力可用。

### 3.5 版本范围与优先级
- 本文档版本分层采用 MVP、V1、V2 三层管理。
- 优先级采用 P0、P1、P2，其中 P0 为上线阻断项，P1 为版本核心项，P2 为增强项。
- MVP 范围冻结后，新增需求默认进入 V1 或 V2，不得直接并入 MVP。
- 需求排期、资源评估与验收均以“版本+优先级+需求ID”作为统一口径。

### 3.7 业务边界与排除项
| 分类 | 本期包含 | 本期不包含 | 延后原因 |
| --- | --- | --- | --- |
| 用户体系 | 注册、登录、会话、资料查询、账户归属校验 | 企业组织、多租户、复杂权限编排 | 当前以个人交易场景为主 |
| 证券交易 | 登录、资产、持仓、下单撤单、基础流水 | 条件单、算法单、两融、衍生品交易 | 依赖通道能力与合规评估 |
| 数字资产 | 密钥校验、账户查询、普通订单 | 套利组合、跨交易所统一清算 | 涉及高风险策略与复杂风控 |
| 策略能力 | 生命周期管理、核心策略类型 | 自定义脚本策略市场、回测平台 | 需独立产品线支撑 |
| 交易计划 | 计划创建、版本管理、评审发布、校验与复盘 | 自动交易全托管执行、跨日自动闭环编排 | 第一阶段聚焦人工主导闭环 |
| 智能能力 | 问答、模型配置、OCR识别 | 自动交易决策托管、投资建议保证收益 | 合规和风控要求高 |

### 3.8 核心术语表
| 术语 | 定义 | 统一口径 |
| --- | --- | --- |
| 策略额度 | 用户可同时运行的策略数量上限 | 按用户等级配置；达到上限后禁止新建运行中策略 |
| 积分占用 | 策略运行期间冻结的积分值 | 占用值随策略盈亏规则动态调整，释放后回到可用积分 |
| 可用积分 | 用户当前可用于新策略的积分值 | 可用积分=总积分-已占用积分 |
| 交易账户 | 用户绑定后可用于交易的账户实体 | 所有交易行为必须校验账户归属关系 |
| 交易所会话 | 已完成认证并可执行交易动作的会话上下文 | 会话失效后必须阻断交易并返回可读原因 |
| 默认交易所会话 | 未显式指定时默认使用的交易所会话 | 客户端可设置默认值，后端按会话标识校验有效性 |
| 场内策略 | 面向股票、ETF、数字货币等可连续撮合资产的策略类型集合 | 统一纳入场内策略口径管理，不再按资产类型拆分网格策略 |
| 场外策略 | 面向基金等非连续撮合资产的策略类型集合 | 统一纳入场外策略口径管理，按申赎与定投规则执行 |
| 计划身份 | 交易计划在同交易日同标的维度下的唯一业务标识 | 同日同标的只允许存在唯一计划身份 |
| 当前版本 | 计划对外生效使用的版本标识 | 单计划同一时刻仅允许一个 current 版本 |
| 发布可用性检查 | 发布前对评审结论、校验状态、阻塞项进行的一次性核验 | 不满足检查条件时必须阻断发布并返回阻塞原因列表 |
| 影响力打分 | 对资讯对资产价格影响程度的评分 | 采用 0-100 分区间，分值越高影响越大 |
| 可用缓存 | 外部依赖失败时可回退的数据副本 | 必须标注缓存时间戳与数据状态 |
| 核心链路 | 登录、行情查询、下单/撤单、策略操作等主流程 | 核心链路失败必须返回错误码与可读原因 |

### 3.9 前端文档中心供给边界说明
- 后端本期不提供前端文档中心内容的数据服务与内容管理接口。
- 前端文档中心由前端配置或静态服务承接。
- 前端若先行上线文档中心，仅可承诺静态展示能力，不得声明后端动态供给能力。
- 如后续需要后端供给文档中心数据，须新增独立需求ID并按版本计划纳入评审与验收。

### 3.11 策略状态机口径
- 策略状态最小集合：待启动、运行中、已停止、已下线、已归档。
- 策略删除定义为业务下线，执行删除动作后状态应进入已下线，不执行物理清除。
- 仅已下线策略可进入已归档状态；归档后不允许恢复为运行中。
- 已停止状态可恢复启动；已下线和已归档状态不允许直接启动。
- 前端与CLI必须使用同一状态语义，不得在客户端重定义删除语义。

## 4. 详细功能需求

### 4.1 用户与会话管理
- 需求ID范围：USR-001~USR-003

#### 4.1.1 注册与登录
- 支持用户注册，完成基础信息合法性校验与重复用户校验。
- 支持用户登录，登录成功后生成有效会话并返回用户基础资料。
- 支持登录态校验，后续业务请求可正确识别用户身份。
- 支持退出登录，退出后当前会话立即失效。

#### 4.1.2 用户资料与状态
- 支持查询用户资料，至少包含等级、额度、积分、已绑定交易账户信息。
- 支持基于用户等级计算策略可用额度。
- 支持基于策略盈亏动态计算积分占用与可用积分。
- 支持按会话查询用户可使用的交易账户集合。

#### 4.1.3 交易账户关系
- 支持识别并维护用户与交易账户绑定关系。
- 支持交易账户解绑。
- 所有交易类操作必须先校验账户归属关系。

### 4.2 资产行情与指标服务
- 需求ID范围：MKT-001~MKT-004

#### 4.2.1 实时行情
- 支持股票实时价格查询。
- 支持基金实时净值查询。
- 支持股票五档盘口查询。
- 支持证券代码、名称等多条件关键字搜索。

#### 4.2.2 历史行情
- 支持股票历史行情按周期与时间范围查询。
- 支持基金历史净值按时间范围查询。
- 支持历史数据缓存命中、缺口识别与增量补齐。
- 支持历史数据合并、去重、排序与范围截取。

#### 4.2.3 技术指标
- 支持股票均线计算，支持多种均线类型与多数据点输出。
- 支持基金均线计算，支持历史窗口与多点计算。
- 历史样本不足时，必须返回不可计算状态及原因。

#### 4.2.4 数据处理规则
- 支持日期格式归一化与交易周期转换。
- 支持默认查询窗口自动推导。
- 支持网络代理策略配置以适配不同部署环境。

### 4.3 新闻资讯与影响力分析
- 需求ID范围：NEWS-001~NEWS-002
- 支持多数据源新闻快讯聚合。
- 支持新闻条目按条件统计。
- 支持新闻影响力打分提交。
- 支持影响力排序起点计算。
- 支持影响力列表分页查询。
- 资讯能力必须支持缓存，避免重复抓取。

### 4.4 证券交易能力
- 需求ID范围：SEC-001~SEC-005

#### 4.4.1 登录与状态
- 支持验证码获取。
- 支持手动登录与自动登录。
- 支持交易所会话状态查询。
- 登录失败时必须返回明确失败原因。

#### 4.4.2 账户资产
- 支持综合资产查询。
- 支持股票持仓查询。
- 支持基金持仓查询。

#### 4.4.3 下单与撤单
- 支持股票下单、基金下单、股票撤单。
- 下单与撤单必须返回业务执行结果及失败原因。

#### 4.4.4 交易流水
- 支持股票委托明细查询。
- 支持股票当日委托查询。
- 支持股票当日成交查询。
- 支持股票未完成委托查询。
- 支持基金历史成交查询。

#### 4.4.5 申购
- 支持新股申购。
- 支持新债申购。

### 4.5 数字资产交易能力
- 需求ID范围：CRY-001~CRY-004

#### 4.5.1 连接与账户
- 支持密钥有效性校验。
- 支持币种持仓查询。
- 支持币种余额查询。

#### 4.5.2 行情与分析
- 支持币价查询。
- 支持历史行情查询。
- 支持均线指标计算。
- 支持资金费率查询。
- 支持成交流水查询。

#### 4.5.3 普通订单
- 支持订单创建、修改、撤销。
- 支持订单状态查询与待成交订单查询。

#### 4.5.4 场内策略交易接入（数字资产）
- 支持为场内策略提供数字资产交易通道接入能力。
- 支持场内策略所需的订单创建、修改、撤销与状态查询。
- 支持场内策略执行所需的最小投入与下单量测算能力。
- 数字资产网格不单独定义策略类型，统一归入场内策略类型管理。

### 4.6 策略管理与执行引擎
- 需求ID范围：STR-001~STR-004

#### 4.6.1 策略生命周期
- 支持策略创建、列表查询、详情查询。
- 支持策略配置修改、运行更新、停止、重启。
- 支持策略完结归档状态管理。

#### 4.6.2 策略类型
- 支持场内策略：覆盖股票、ETF、数字货币，包含统一网格、马丁、动态平衡、订单队列等策略能力。
- 支持场外策略：覆盖基金，包含动态平衡、定投等策略能力。

#### 4.6.3 策略计算能力
- 支持场内网格价格带、订单生成与成交后重建。
- 支持场内网格资产占用、投资需求、限价能力测算。
- 支持平衡策略可用现金测算与调仓单生成。
- 支持年化收益计算。
- 支持定投触发条件校验与仓位管理计算。

#### 4.6.4 策略公共校验
- 所有策略操作必须校验登录状态。
- 所有策略操作必须校验令牌合法性。
- 所有策略操作必须校验请求主体一致性。
- 所有策略操作必须校验可用策略额度。
- 所有策略操作必须校验交易账户归属关系。

#### 4.6.5 策略状态流转约束
- 创建成功后的初始状态可为待启动或运行中，需按策略类型统一规则返回。
- 停止操作必须使运行中策略进入已停止状态。
- 删除操作必须使策略进入已下线状态。
- 归档操作仅允许作用于已下线策略，归档后状态为已归档。
- 查询接口必须返回标准化状态字段，保障前端与CLI统一展示。

### 4.7 统一交易抽象层
- 需求ID：UTL-001
- 必须提供统一交易能力模型，屏蔽不同交易通道差异。
- 必须提供交易通道适配器注册机制，支持后续扩展。
- 策略层必须通过统一抽象访问交易行为，不得直接依赖具体通道实现。
- 统一抽象能力需覆盖下单、撤单、查单、持仓、成交等核心动作。

### 4.8 智能问答与工具调用
- 需求ID范围：AI-001~AI-002、OCR-001

#### 4.8.1 智能问答
- 支持普通问答与投资场景问答。
- 支持结合策略、股票、基金上下文增强回答质量。
- 支持模型设置配置化管理，参数可持久化与读取。
- 支持模型列表查询供前端动态展示。

#### 4.8.2 OCR 与辅助能力
- 支持验证码图片识别。
- 识别结果必须返回标准状态与可读错误信息。

#### 4.8.3 工具扩展
- 支持多工具协同调用，满足复杂问题分步处理。
- 支持后续新增工具接入且不影响主流程稳定性。

### 4.9 应用入口与基础能力
- 需求ID：APP-001
- 提供统一应用入口，集中注册业务能力。
- 提供统一日志初始化能力。
- 提供统一响应清洗能力，减少无效空字段输出。

### 4.10 交易计划能力
- 需求ID范围：PLN-001~PLN-006

#### 4.10.1 计划主流程能力
- 支持交易日维度计划列表查询，支持按交易日、状态、市场、标的关键字筛选。
- 支持计划详情查询，返回计划身份、当前版本、关键位、状态、最新风控结论与校验状态。
- 支持计划创建，满足同交易日同标的唯一计划身份约束并完成初始版本初始化。
- 支持在草稿或评审中状态下更新版本业务字段。
- 支持版本下 1..N 情景批量替换与排序重建。
- 支持基于当前版本创建下一版本，用于已批准或已发布计划修订。

#### 4.10.2 状态机与评审发布能力
- 支持草稿、评审中、已批准、已发布、已归档状态流转。
- 状态迁移必须先记录状态日志再执行状态变更。
- 支持风险评审与复核记录，包含结论、风险等级、仓位决策、执行要点。
- 支持发布可用性检查，返回评审就绪、校验就绪和阻塞原因列表。
- 支持发布提交，发布动作需同步写入发布记录、状态日志、状态更新。
- 发布前必须满足最新风险评审有效且最新校验通过并命中当前内容版本。

#### 4.10.3 校验与质量能力
- 支持手工触发校验，按目标版本执行规则并记录运行结果。
- 支持校验问题明细分级输出，包含错误、警告、提示。
- 支持最新校验结果与校验历史查询，满足发布页和质量看板使用。
- 支持质量概览查询，按时间范围统计通过率、问题分布、规则命中情况。
- 校验结果必须绑定版本内容更新时间，避免旧校验结果误用于新内容发布。

#### 4.10.4 执行记录与复盘能力
- 支持信号触发记录，用于记录情景触发但未成交场景。
- 支持执行订单记录，并校验计划、版本、情景、交易上下文一致性。
- 支持执行订单幂等保护，同一交易上下文重复订单引用不得重复写入。
- 支持持仓快照与权益快照写入，满足日终复盘与仓位核验。
- 支持复盘聚合查询，汇总版本演进、信号、订单、持仓、权益与执行偏差摘要。

#### 4.10.5 数据约束能力
- 交易计划数据写入以数据库为唯一事实来源。
- 百分比、金额、价格统一采用整数口径存储，保证跨端口径一致。
- 已批准、已发布、已归档版本禁止原地改写业务内容。
- 需具备关键数据约束能力，覆盖单计划单 current 版本、合法状态迁移、发布前置条件、金额一致性。
- 删除策略以逻辑归档为主，计划、评审、执行、发布链路默认禁止物理级联删除。

## 5. 非功能需求

### 5.1 性能
- 高频查询场景必须通过缓存降低外部依赖压力，MVP 阶段缓存命中率目标不低于 70%。
- 行情、资产、策略列表等查询接口在正常负载下，P95 响应时间不高于 1.5 秒。
- 外部源失败时应优先返回可用缓存并标注状态，缓存回退命中响应时间不高于 800ms。

### 5.2 安全
- 必须具备恶意请求拦截策略，注入与越权类高风险请求拦截率目标为 100%。
- 必须具备鉴权失败快速返回能力，未授权请求应在 200ms 内返回失败结果。
- 必须遵循敏感信息脱敏与最小暴露原则，日志中密钥、令牌、密码字段脱敏率为 100%。

### 5.3 可维护性
- 业务模块边界清晰、职责单一，核心模块必须可独立发布与回滚。
- 公共能力统一复用，避免重复实现，跨模块公共能力重复建设率应控制在 10% 以下。
- 新增交易通道与策略类型不影响既有业务能力，新增通道不应改动既有策略调用方契约。

### 5.4 可观测性
- 关键业务动作必须记录日志，核心链路关键节点日志覆盖率不低于 95%。
- 异常必须记录上下文并支持追踪，错误日志需包含请求ID、用户标识、模块标识、错误码。
- 支持按模块定位故障来源，故障初步定位时间目标不超过 30 分钟。

## 7. 风险与前提假设
### 7.1 关键前提
- 外部行情、资讯、交易通道在交易时段具备基础可用性。
- 第三方通道协议在版本周期内无破坏性变更。
- 合规边界允许当前需求范围内的查询、交易与问答能力交付。

### 7.2 主要风险
- 外部依赖抖动可能导致实时数据延迟或下单失败率上升。
- 多通道接入差异可能影响统一抽象层交付节奏。
- 智能问答结果稳定性受模型服务波动影响。
- 合规要求调整可能导致部分能力延期。

### 7.3 风险应对原则
- 对外部依赖提供缓存回退、重试与明确状态标识。
- 对核心通道采用灰度接入与回滚预案。
- 对合规敏感能力设置开关并预留降级路径。
